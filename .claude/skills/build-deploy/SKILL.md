# Build and Deploy Pipeline

## Toolchain Overview

All tools are managed by **Rokit** (`rokit.toml`) and installed with `rokit install`:

| Tool       | Version    | Purpose                                    |
|------------|------------|--------------------------------------------|
| Rojo       | 7.7.0-rc.1 | Syncs Luau source files into Roblox place files |
| Wally      | 0.3.2      | Package manager for Roblox libraries        |
| Lune       | 0.10.4     | Luau script runner (used for the build merge script) |
| Mantle     | 0.11.18    | Infrastructure-as-code deployment to Roblox |
| StyLua     | 0.20.0     | Luau/Lua code formatter                     |
| Selene     | 0.27.1     | Luau/Lua linter                             |

## Project Structure (Rojo)

The `default.project.json` maps local directories into the Roblox DataModel:

```
src/server/     -> ServerScriptService.Server     (server scripts)
src/client/     -> StarterPlayer.StarterPlayerScripts (client scripts)
src/shared/     -> ReplicatedStorage.Shared        (shared modules)
src/test/       -> ServerScriptService.TestRunner   (test runner)
Packages/       -> ReplicatedStorage.Packages       (Wally shared packages)
ServerPackages/ -> ServerScriptService.ServerPackages (Wally server packages)
DevPackages/    -> TestService.DevPackages          (Wally dev packages like TestEZ)
```

Rojo only manages **scripts and packages**. It does not contain the map, workspace parts, lighting, or other Studio-authored content. Those live in the base place file (`places/base.rbxlx`), which is edited in Roblox Studio and committed to git.

## The Build System

### Why a merge step exists

Rojo builds scripts into a place file, but it cannot represent hand-built Studio content (terrain, models, lighting, etc.). The base place file `places/base.rbxlx` contains all that Studio-authored content. The Lune build script merges them together into a single deployable place file.

### How it works

The build pipeline runs through `make build` (or `make build-dev` / `make build-prod`):

1. **`scripts/build.sh`** orchestrates the build:
   - Generates `src/shared/ProductIds.lua` from Mantle state (maps product names to Roblox asset IDs)
   - Generates `src/shared/BadgeIds.lua` from Mantle state (maps badge names to Roblox badge IDs)
   - Calls `lune run scripts/build` to run the merge script

2. **`scripts/build.luau`** (the Lune merge script) performs the actual merge:
   - Runs `rojo build default.project.json -o build/scripts.rbxlx` to build scripts
   - Loads `places/base.rbxlx` (the Studio-authored base place)
   - Loads the Rojo output (`build/scripts.rbxlx`)
   - Merges services: ReplicatedStorage, ServerScriptService, StarterPlayer, TestService
   - For folders that exist in both, it recursively merges (Rojo content takes precedence for scripts)
   - Outputs `build/Game.rbxlx` (the final deployable place)

### Build commands

```bash
make build          # Build for dev (default)
make build-dev      # Explicit dev build
make build-prod     # Build for prod environment
```

The environment argument (`dev` or `prod`) only affects which Mantle state is used for generating ProductIds and BadgeIds. Pass `prod` when building for production deployment.

### Auto-generated files

These files are generated by the build and should not be edited manually:

- `src/shared/ProductIds.lua` - Generated by `scripts/generate-product-ids.sh`
- `src/shared/BadgeIds.lua` - Generated by `scripts/generate-badge-ids.sh`

Both scripts parse `.mantle-state.yml` (created by Mantle after deployment) to extract the Roblox-assigned asset IDs for products and badges.

## Wally (Dependency Management)

Dependencies are declared in `wally.toml`:

```toml
[dependencies]         # Shared packages -> Packages/
Knit = "sleitnick/knit@1.6.0"
GoodSignal = "sleitnick/signal@2.0.0"
Promise = "evaera/promise@4.0.0"
Fusion = "elttob/fusion@0.3.0"
OnyxUI = "imavafe/onyx-ui@1.0.2"
TopbarPlus = "1foreverhd/topbarplus@3.4.0"

[server-dependencies]  # Server-only -> ServerPackages/
ProfileService = "thunn/profileservice@3.2.1"

[dev-dependencies]     # Dev/test only -> DevPackages/
TestEZ = "roblox/testez@0.4.1"
```

### Adding or updating a package

1. Edit `wally.toml` to add/change the dependency
2. Run `wally install` to download packages into `Packages/`, `ServerPackages/`, or `DevPackages/`
3. The Rojo project maps these directories into the DataModel automatically

Wally uses the UpliftGames registry at `https://github.com/UpliftGames/wally-index`. Search for packages there or at `https://wally.run`.

## Mantle (Deployment)

Mantle manages the Roblox experience as infrastructure-as-code. Configuration lives in `mantle.yml`.

### Environments

| Environment | Access   | Notes                                          |
|-------------|----------|------------------------------------------------|
| dev         | private  | Name prefixed with environment label, free private servers, Studio API access enabled |
| prod        | public   | All device types including console, paid private servers ($100), Studio API access disabled |

### What Mantle manages

- **Experience configuration**: playability, avatar settings, genre
- **Place file**: uploads `build/Game.rbxlx` as the start place
- **Developer products**: all in-game purchases
- **Badges**: achievement badges
- **Icon and thumbnails**: game icon and thumbnail images from `assets/`

### Deployment commands

```bash
make deploy-dev       # Deploy to dev environment
make deploy-prod      # Deploy to prod environment
make deploy-dry-run   # Dry run against dev (shows what would change without applying)
make deploy           # Alias for deploy-dev
make mantle-outputs   # Show current Mantle resource outputs (IDs, URLs)
```

### Deployment prerequisites

- `ROBLOSECURITY` environment variable must be set (Roblox auth cookie)
- Run `make build` (or `make build-dev` / `make build-prod`) first to produce `build/Game.rbxlx`
- Mantle state is stored locally in `.mantle-state.yml` (not committed to git)

### Deployment workflow

```bash
# Typical dev deploy:
make build-dev && make deploy-dev

# Typical prod deploy:
make build-prod && make deploy-prod

# Check what would change without deploying:
make deploy-dry-run
```

### Adding a new developer product

1. Add the product definition to `mantle.yml` under `target.experience.products`
2. Deploy to create the product on Roblox: `make deploy-dev`
3. Rebuild to regenerate ProductIds.lua: `make build-dev`
4. Reference the product in Lua code via `ProductIds["your-product-name"]`

### Adding a new badge

1. Add the badge definition to `mantle.yml` under `target.experience.badges`
2. Provide an icon image (currently using `assets/badges/placeholder-badge.png`)
3. Deploy: `make deploy-dev`
4. Rebuild to regenerate BadgeIds.lua: `make build-dev`
5. Reference in Lua code via `BadgeIds["your-badge-name"]`

## Linting and Formatting

### StyLua (Formatter)

Configuration in `stylua.toml`:
- 100 column width
- 2-space indentation (spaces, not tabs)
- Unix line endings
- Double quotes preferred
- Parentheses on all calls

```bash
make format         # Auto-format all files in src/
make format-check   # Check formatting without modifying (used in CI)
```

### Selene (Linter)

Configuration in `selene.toml`:
- Uses the `roblox` standard library
- Shadowing: warn
- Unused variables/functions: warn
- Incorrect standard library use: deny (error)

```bash
make lint           # Run Selene on src/
```

### Fixing CI lint/format failures

If CI fails on formatting:
```bash
make format         # Auto-fix formatting
```

If CI fails on linting, read the Selene output and fix the issues manually. Common issues:
- Unused variables: prefix with `_` or remove them
- Shadowing: rename the inner variable
- Incorrect API use: check the Roblox API docs for the correct method signature

## CI/CD Pipeline

The GitHub Actions workflow (`.github/workflows/ci.yml`) runs on every push and pull request:

1. **Checkout** code
2. **Install Rokit** (which provides all the tools)
3. **Format check** (`stylua --check src`) - fails if code is not formatted
4. **Lint** (`selene src`) - fails if lint errors are found
5. **Rojo build** - verifies the project compiles (builds to `build/MyGame.rbxlx`)

Note: CI does not run the full Lune merge build or deploy. It only validates that code is clean and Rojo can build the project. Deployment is done manually via `make deploy-dev` or `make deploy-prod`.

## Common Workflows

### "I changed code, how do I validate it?"

```bash
make format         # Fix formatting
make lint           # Check for lint issues
make build          # Build the full place file
```

### "I need to test in Roblox Studio"

Use `make serve` to start a Rojo live-sync server. Connect to it from within Studio using the Rojo plugin. This gives you live reloading of script changes but does not include the base place content.

For a full test with the base place, run `make build` then open `build/Game.rbxlx` in Studio.

### "I added a new developer product or badge"

```bash
# 1. Edit mantle.yml to add the product/badge
# 2. Deploy to create it on Roblox
make deploy-dev
# 3. Rebuild to pick up the new IDs
make build-dev
# 4. Use ProductIds["name"] or BadgeIds["name"] in your Lua code
```

### "How do I deploy to production?"

```bash
make build-prod     # Build with prod IDs
make deploy-prod    # Deploy to the public experience
```

### "CI is failing, what do I do?"

1. Read the CI output to identify which step failed
2. If format check failed: run `make format` and commit
3. If lint failed: run `make lint`, read the errors, fix them
4. If Rojo build failed: check `default.project.json` paths match actual files in `src/`
