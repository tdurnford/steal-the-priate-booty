  Follows GearController pattern: loads initial state from DataService:GetData(), listens to
  CosmeticService.CosmeticChanged for server-pushed updates, listens to DataService.DataChanged
  for reconnection sync. Maintains local OwnedCosmetics and EquippedCosmetics caches. Public
  API: OwnsCosmetic, GetEquippedInSlot, GetEquippedCosmetics, GetOwnedCosmetics,
  PurchaseCosmetic, EquipCosmetic, UnequipCosmetic, GetCosmeticCatalog. Local signal:
  CosmeticChanged for other controllers. Plays purchase SFX via SoundController.
- No Types.lua changes needed — ownedCosmetics, equippedCosmetics, EquippedCosmetics type
  already exist with correct fields.
- No DataService changes needed — all required methods (OwnsCosmetic, AddOwnedCosmetic,
  EquipCosmetic, UpdateTreasury) already implemented.

## 2026-03-02 — HAZARD-004: Rogue Wave Hazard (Night Only)

### What was done
- Created `src/server/Services/RogueWaveService.lua` — Server-authoritative night-only rogue wave
  hazard. Loads Parts from workspace.RogueWaveZones folder. Each Part's LookVector defines the
  inland push direction. Night-only activation: listens to DayNightService.PhaseChanged, schedules
  1-2 waves per night at random times (minimum 15s spacing). Wave lifecycle: idle → warning (6s)
  → impact (2s) → recede (3s) → idle. During impact phase, continuously checks players in zone
  (AABB detection). Hit players: 3.0s ragdoll via SessionStateService, pushed 20-25 studs inland
  with upward scatter, 20% loot spill via GameConfig.calculateSpill + DoubloonService:ScatterDoubloons.
  Displaces loose pickups inland via DoubloonService:DisplacePickupsInRegion. After each wave
  recedes: spawns 2-3 bonus containers (50% Crate, 35% Barrel, 15% Treasure Chest) at random
  positions within the zone via ContainerService:SpawnContainerAt. Skips: ragdolled, dashing
  (i-frame bypass), tutorial, harbor. Client signals for VFX/SFX. Server signal
  PlayerHitByWave for inter-service communication. Night end: cancels scheduled waves, resets
  active zones to idle. Disconnect cleanup.
- Created `src/client/Controllers/RogueWaveController.lua` — Client-side rogue wave VFX/SFX.
  Warning phase: water wall rises from zone edge (Glass material tween), deep roaring SFX
  (pitched down for ominous effect, volume ramps up), screen-edge water vignette overlay
  (UIGradient for transparent center, blue edges). Impact phase: wave crashes down (fast
  exponential tween), massive spray + foam particle emitters, glow PointLight, loud wave crash
  SFX, screen overlay flash. Recede phase: water retreats and fades. "ROGUE WAVE!" notification
  on hit. Physics ragdoll via RagdollModule with push velocity. "The wave washed treasure
  ashore!" notification for nearby players when bonus containers spawn. Late-join sync via
  GetZoneStates RPC. Character removal cleanup.
- GameConfig.RogueWave already existed with correct values (perNightCount 1-2, warningDuration 6s,
  ragdollDuration 3.0s, pushDistance 20-25, lootSpillPercent 0.20, bonusContainerCount 2-3).
- No Types.lua changes needed — reuses existing ragdoll/doubloon SessionState fields.

### Acceptance criteria met
- 1-2 rogue waves per night cycle (random timing with 15s minimum spacing) ✓
- 6-second warning with roaring SFX, screen-edge water VFX, and ocean visibly rising ✓
- Players caught: 3.0s ragdoll, 20-25 stud push inland, 20% spill ✓
- 2-3 bonus containers wash ashore after each wave ✓
- Night-only; does not occur during day (PhaseChanged listener, dawn cancels all) ✓
- Server-authoritative (all detection, ragdoll, spill on server) ✓

### Design decisions
- **Event-driven night scheduling**: Instead of continuous cycling like TidalSurge, RogueWave
  listens to DayNightService.PhaseChanged and schedules 1-2 waves at random times during each
  night. Waves are spaced at least 15s apart to prevent stacking. On dawn, all pending waves
  are cancelled and active zones reset to idle.
- **Separate zone folder**: Uses workspace.RogueWaveZones (not TidalSurgeZones) to allow map
  designers to place rogue wave zones on exposed coastlines independently from tidal surge zones.
- **Stronger effects than TidalSurge**: Rogue waves are rarer (1-2/night vs continuous) but
  more impactful: 3.0s ragdoll (vs 1.5s), 20-25 stud push (vs 10-15), 20% spill (vs 10%),
  and guaranteed 2-3 bonus containers (vs 25% chance of 1).
- **Screen overlay vignette**: Unlike tidal surge, rogue wave uses a full-screen blue vignette
  overlay during warning phase to alert nearby players. The overlay uses UIGradient for a
  subtle transparent-center effect and flashes brighter on impact.
- **Idle at math.huge**: Zones stay idle with phaseTimer = math.huge until explicitly triggered
  by the night scheduling system, preventing any unintended cycling during day.

## 2026-03-02 — HAZARD-003: Quicksand Hazard

### What was done
- Created `src/server/Services/QuicksandService.lua` — Server-authoritative quicksand hazard.
  Loads Parts from workspace.QuicksandPatches folder. Maintains 2-3 active patches out of 4-6
  total via cycling system (each active patch stays active 25-40s, then goes dormant and a
  random dormant patch activates). Player position checks every 0.25s on active patches (AABB).
  Players stepping on active quicksand: 3s immobilization (WalkSpeed=0, JumpHeight=0, can still
  attack and pickup). After 3s: ejected to nearest patch edge. Stores/restores WalkSpeed and
  JumpHeight for correct restoration. Ragdoll from other sources frees the player immediately.
  Skips: dashing (i-frame bypass), ragdolled, tutorial, harbor. Client signals for VFX/SFX.
  Server signals for inter-service communication. Disconnect cleanup.
- Created `src/client/Controllers/QuicksandController.lua` — Client-side quicksand VFX/SFX.
  Active patches show subtle sandy shimmer particles + occasional bubble particles. Dormant
  patches have no visual effects (normal ground). On trap: sand swirl VFX part at player's feet,
  sinking sound, notification. On release: pop-out sound, freed notification, VFX cleanup.
  Late-join sync via GetPatchStates RPC. Character removal cleanup.
- Modified `src/shared/Types.lua` — Added `isQuicksandTrapped: boolean` and
  `quicksandEndTime: number` to SessionState type + defaults.
- Modified `src/shared/GameConfig.lua` — Expanded Quicksand config with `activeDurationMin` (25s),
  `activeDurationMax` (40s), and `ejectOffset` (5 studs).
- Modified `src/server/Services/SessionStateService.lua` — Added quicksand methods:
  `IsQuicksandTrapped(player)` with auto-clear on expiry, `StartQuicksandTrap(player, duration)`,
  `EndQuicksandTrap(player)`. All fire `notifyChange` signals.

### Acceptance criteria met
- 4-6 patches loaded from workspace.QuicksandPatches; 2-3 active at any time ✓
- Active patches have subtle visual tell (shimmer particles, bubble particles) ✓
- Player is immobilized for 3s, can still attack but not move ✓
- Ejected to nearest solid ground (nearest patch edge + offset) after 3s ✓
- Dash i-frames (0.3s) prevent quicksand trigger ✓
- Server-authoritative (all detection and immobilization on server) ✓

### Design decisions
- **Cycling mechanism**: Each active patch has its own timer (25-40s randomized). When it
  expires, the patch goes dormant and a random dormant patch activates. This maintains the
  2-3 active count while providing unpredictable hazard distribution.
- **WalkSpeed/JumpHeight save-restore**: Rather than querying RankEffectsService or CombatService
  for the "correct" speed, we save the player's current WalkSpeed and JumpHeight at trap time
  and restore them on release. This correctly handles any modifiers (rank speed bonus, block
  speed reduction) that were active when the player stepped in.
- **Ragdoll frees quicksand**: If a player gets ragdolled by another source while trapped (e.g.,
  volcanic vent, combat), the quicksand trap ends immediately. Movement is restored but no
  ejection teleport occurs (ragdoll physics handles positioning). This prevents double-stacking
  movement lockouts.
- **0.25s position check interval**: Unlike volcanic vents (per-frame), quicksand only checks
  player positions every 0.25s. This is sufficient since quicksand doesn't need frame-precise
  detection and saves CPU with 4-6 patches × all players.
- **No loot spill**: The spec explicitly says "no ragdoll, no spill, but vulnerable". Players
  trapped in quicksand keep their doubloons but are sitting ducks for PvP attacks.
- **AABB detection**: Matches VolcanicVentService and TidalSurgeService pattern using
  PointToObjectSpace for rotated patch Parts.
- **Safe iteration**: Collected players-to-release into temporary arrays before modifying
  TrappedPlayers table to avoid issues with table modification during iteration.

### Notes for next developer
- QuicksandPatches folder: Place Parts in workspace.QuicksandPatches to define quicksand zones.
  Each Part defines a quicksand patch boundary. If the folder is empty, quicksand is disabled.
- Patches should be placed on jungle paths between Pirate Town and Skull Cave, and along jungle
  trails (per MAP-005 spec).
- The eject position is calculated as the nearest patch edge + 5 studs outward. The player is
  teleported via CFrame to that position.
- Sound asset IDs are shared placeholders (same as volcanic vent/tidal surge). Replace with
  dedicated quicksand sounds when AUDIO-001 is implemented.
- Next priorities: HAZARD-004 (rogue wave), UI-005 (minimap), UI-006 (event banner queuing),
  COSMETIC-001/002/003 (cosmetic system)

## 2026-03-02 — NPC-009: Night NPC Behavior Modifiers

### What was done
- Added skeleton pack hunting AI to `src/server/Services/NPCService.lua`. At nightfall,
  nearby budget skeletons pair up into packs of 2. When one pack member aggros a player,
  it alerts its partner to chase the same target. The flanking skeleton paths to an offset
  position perpendicular to the leader-to-target vector instead of directly to the target.
- Added pack hunting config to `src/shared/GameConfig.lua` (formation radius, flank offset,
  max packs).
- The +25% night speed bonus and +30% night aggro range bonus were already implemented in
  `getEffectiveSpeed()` and `getEffectiveAggroRange()` functions (from prior work). This
  feature adds the pack coordination layer on top.

### Pack formation system
1. **Night transition**: `formSkeletonPacks()` called during Dusk/Night phase. Collects all
   alive, non-bonus, non-dormant budget skeletons. Sorts by zone. Pairs closest skeletons
   within `packFormationRadius` (60 studs). Capped at `packMaxPacks` (5) simultaneous packs.
2. **Pack structure**: Each pack has a leader and a flanker. Leader is the first to aggro.
   Flanker is assigned `isPackFlanker = true`.
3. **Dawn cleanup**: `dissolveAllPacks()` clears all pack fields on all NPCs.

### Pack chase behavior
1. **Target sharing**: When a pack skeleton enters CHASE from patrol, `alertPackPartner()`
   wakes the partner (if in patrol/idle/loot_pickup) and sets it to CHASE the same player.
2. **Flanking**: During chase pathfinding, flankers compute a flank position: perpendicular
   offset (`packFlankOffset` = 10 studs) from the leader-to-target line. This positions them
   on the opposite side of the target from the leader.
3. **Target stickiness**: Pack members don't re-evaluate for closer targets (normal skeletons
   switch to a 60% closer target). Pack members stick to their shared target.
4. **Attack**: Once in attack range, flankers attack normally (slash/lunge). The coordination
   is in approach direction, not attack timing.

### Pack lifecycle handling
- **Death cleanup**: When a pack member is killed (`KillNPC`), partner's pack fields are
  cleared and `ActivePackCount` decremented. Partner becomes a solo skeleton.
- **Despawn cleanup**: `despawnNPC` also clears partner pack fields.
- **Stale check**: `formSkeletonPacks()` dissolves any stale packs before forming new ones.
- **Partner validation**: `alertPackPartner()` validates partner is alive and shares the
  same packId before alerting. If invalid, dissolves the broken pack.

### Config additions (`GameConfig.NPCBehavior`)
- `packFormationRadius = 60` — max distance for pairing
- `packFlankOffset = 10` — perpendicular offset for flanker pathfinding
- `packMaxPacks = 5` — max simultaneous packs

### NPCEntry new fields
- `packId: number?` — shared pack identifier (nil if solo)
- `packPartnerId: number?` — NPC ID of pack partner
- `isPackFlanker: boolean` — true for flanking skeleton, false for leader

### Night modifier summary (all acceptance criteria met)
- All NPCs +25% speed at night — already in `getEffectiveSpeed()` via `nightSpeedBonus`
- All NPCs +30% aggro range at night — already in `getEffectiveAggroRange()` via `nightAggroRangeBonus`
- Skeletons form packs and coordinate via flanking — new pack hunting system
- All modifiers revert at Dawn — speed/aggro are dynamic (check IsNight each tick),
  packs dissolved by `dissolveAllPacks()`

### Design decisions
- **Only budget skeletons pack**: Bonus NPCs (from ThreatEffectsService) and dormant NPCs
  are excluded from pack formation. They have specialized behavior already.
- **Forward-declared alertPackPartner**: Uses `local alertPackPartner` forward declaration
  (defined after `setState`) because it needs `setState()` which is defined later in the file.
- **Max 5 packs**: Limits pack coordination overhead. With 9-15 night skeletons, this means
  5 packs (10 paired) + some solo skeletons for variety.
- **Flanking via perpendicular offset**: Simple but effective. The flanker approaches from
  the side instead of directly behind the leader. Distance-based, not angle-based.
- **Ghost Pirates and Phantom Captains don't pack**: Ghost Pirates are ambush predators
  (short 15-stud aggro). Phantom Captains hunt specific players. Pack fields exist on their
  entries for type consistency but are always nil.

### Notes for next developer
- `alertPackPartner` uses a forward declaration pattern. If you need to add more pack
  behavior that calls setState, add it near `alertPackPartner` (after setState definition).
- Pack formation only happens at Dusk/Night transition. Newly spawned skeletons mid-night
  won't automatically join existing packs.
- The flank offset is always to the left (perpendicular rotation). For more varied flanking,
  you could alternate sides based on NPC ID parity.

  UI-005 (minimap), TUTORIAL-001/002

## 2026-03-02 — NPC-002: Cursed Skeleton Loot Pickup Behavior

### What was done
- Extended `src/server/Services/NPCService.lua` with loot pickup AI behavior for
  Cursed Skeletons. Skeletons now detect and collect loose doubloon pickups during
  patrol, carry them, and display a coin purse visual that scales with carried amount.
- Extended `src/server/Services/DoubloonService.lua` with `FindNearestPickup()` and
  `CollectPickupsNear()` APIs for NPC-driven pickup collection.
- Added loot pickup config to `src/shared/GameConfig.lua` (scan radius, pickup radius,
  scan interval, carry cap).

### New AI state: LOOT_PICKUP
1. **Scan during patrol**: Every `lootScanInterval` (2s), skeleton scans for loose
   doubloon pickups within `lootScanRadius` (25 studs) using
   `DoubloonService:FindNearestPickup()`.
2. **Path to pickup**: If a pickup is found, skeleton transitions to LOOT_PICKUP state
   and uses SimplePath to navigate to the pickup position.
3. **Collect on arrival**: When within `lootPickupRadius` (3 studs), skeleton calls
   `DoubloonService:CollectPickupsNear()` which removes pickup entities and returns
   their total value. Value is added to `entry.carriedDoubloons`.
4. **Return to patrol**: After collecting (or if blocked/timed out after 8s), returns
   to PATROL state.
5. **Combat priority**: LOOT_PICKUP state checks aggro range each tick. If a player
   enters aggro range, skeleton immediately transitions to CHASE (combat > loot).

### Coin purse visual system
1. **3 tiers**: Small (50+ doubloons, brown), Medium (200+, dark gold with glow),
   Large (500+, gold with glow + shimmer particles).
2. **Server-replicated**: Coin purse is a Part welded to HumanoidRootPart at belt
   offset (-0.8, -0.5, 0.3). Automatically visible to all clients.
3. **Auto-update**: `updateCoinPurseVisual()` called after every collection. Old purse
   destroyed, new tier-appropriate purse created only when tier changes.
4. **PointLight**: Medium/large purses have gold PointLight (range 5-8 studs).
5. **Particles**: Large purse has gold shimmer ParticleEmitter.

### DoubloonService new public APIs
1. **`FindNearestPickup(position, radius)`**: Returns the nearest pickup entry and its
   index within the given radius. Uses squared-distance for performance.
2. **`CollectPickupsNear(position, radius)`**: Removes all pickups within radius,
   destroys their Parts, returns total doubloon value collected.

### Carry cap and overflow handling
- `maxCarriedDoubloons = 500`: Skeleton stops scanning when at cap.
- If `CollectPickupsNear()` returns more than remaining capacity, excess doubloons
  are scattered back at the skeleton's position (prevents value loss).

### SimplePath signal integration
- `Reached` during LOOT_PICKUP: marks arrival time for collection on next tick.
- `Blocked` during LOOT_PICKUP: gives up, clears target, returns to patrol.
- `Error` during LOOT_PICKUP: same as blocked — returns to patrol.

### Client signal added
- `NPCLootPickup(npcId, position, amount, newTotal)` — fired to all players when a
  skeleton collects doubloons. Can be used for pickup SFX/VFX on client.

### What was already implemented (no changes needed)
- `carriedDoubloons` field already existed in NPCEntry.
- `KillNPC()` already drops `carriedDoubloons + bonusDoubloons` on death.
- `despawnAllGhostPirates()` already drops carried doubloons at Dawn.

### Design decisions
- **Only skeletons pick up loot**: Ghost Pirates are spectral ambush predators and
  don't interact with physical objects. `scanForNearbyPickup()` skips non-skeleton NPCs.
- **Scan interval (2s) prevents perf impact**: Only one scan per skeleton every 2 seconds,
  not every frame. The scan uses DoubloonService's O(n) pickup list.
- **8s timeout**: If a skeleton can't reach a pickup within 8 seconds, it gives up and
  returns to patrol. Prevents pathfinding softlocks.
- **Coin purse thresholds match LOOT-006**: Uses the same 50/200/500 tiers planned for
  player loot visibility, keeping visual language consistent.
- **Belt weld offset**: Purse is offset to the left hip, slightly forward — visible from
  most camera angles without clipping into the NPC's body.

### Notes for next developer
- The `NPCLootPickup` client signal is ready for a client-side controller to play
  collection SFX/VFX (e.g., coin jingle, brief gold particle burst at NPC position).
- Coin purse Part uses `Enum.PartType.Ball` + `Enum.Material.Fabric` — replace with
  a proper Mesh when art assets are available.
- The scan radius (25 studs) means skeletons can detect pickups about half their aggro
  range away. This keeps loot pickup as an "opportunistic" behavior during patrol.
- Carry cap (500) ensures a single skeleton doesn't accumulate game-breaking amounts.
  A skeleton carrying 500+ is already a high-value kill target.
- Ghost Pirate NPCEntry has the loot pickup fields (set to 0/nil) for type consistency
  but they are never used by the loot pickup logic.

  NPC-009 (night NPC modifiers)

## 2026-03-02 — NPC-004: SimplePath Chase Behavior

### What was done
- Upgraded NPC chase movement in `src/server/Services/NPCService.lua` from basic
  `Humanoid:MoveTo()` to SimplePath pathfinding with 0.3s recalculation intervals.
- Added stuck detection: 3s triggers path recalculation, 6s triggers teleport
  to nearest patrol waypoint with poof VFX signal.
- Added Harbor boundary waiting: NPC stops at boundary for 5s before returning
  to patrol. If target leaves Harbor during wait, chase resumes.
- Added `NPCTeleported` client signal for poof VFX when NPCs teleport due to
  stuck detection.

### Chase state changes (lines 1180-1331)
1. **SimplePath:Run() replaces Humanoid:MoveTo()**: Chase now uses PathfindingService
   via SimplePath, recalculating every 0.3s (`NPC_BEHAVIOR.chasePathRecalcInterval`).
   This lets NPCs navigate around obstacles instead of walking into walls.
2. **0.3s recalc interval**: `lastChaseRecalcTime` tracks when SimplePath was last
   called. Prevents excessive path recomputation each frame.
3. **SimplePath signal handling**: Reached/Blocked/Error signals now check aiState.
   During chase, they reset `lastChaseRecalcTime = 0` to force immediate recalc
   (e.g., if path is blocked by a new obstacle, NPC immediately recomputes).

### Stuck detection system
1. **Position tracking**: `chaseStuckPos` and `chaseStuckTime` track the NPC's
   position at the start of each stuck-check window.
2. **Movement threshold**: If NPC hasn't moved ≥2 studs (`stuckMoveThreshold`)
   within the window, it accumulates stuck time.
3. **3s recalc**: After 3s stuck, forces path recalculation (resets recalc timer).
4. **6s teleport**: After 6s stuck, calls `teleportToNearestWaypoint()` which
   finds the closest patrol waypoint in the NPC's zone and teleports there.
   Fires `NPCTeleported` signal for client-side poof VFX.
5. **Reset**: When the NPC moves enough, the stuck window resets.

### Harbor boundary wait
1. **5s wait**: When target enters Harbor, NPC stops SimplePath and waits at
   its current position for `harborReturnDelay` (5s).
2. **Resume on exit**: If the target leaves Harbor during the wait, chase resumes
   immediately with a forced path recalculation.
3. **Timeout**: After 5s, NPC gives up chase and returns to patrol.

### setState changes
- `setState()` now stops SimplePath when leaving chase state (previously only
  stopped when leaving patrol).
- Entering chase resets all chase tracking fields: `lastChaseRecalcTime`,
  `chaseStuckPos`, `chaseStuckTime`, `harborWaitStart`.

### Leash return improvement
- When NPC exceeds 80-stud leash, it now uses `SimplePath:Run(spawnPosition)`
  to pathfind back (previously used `Humanoid:MoveTo()` which could get stuck
  on obstacles).

### New NPCEntry fields
- `lastChaseRecalcTime: number` — last time SimplePath:Run() called during chase
- `chaseStuckPos: Vector3?` — position at start of stuck-check window
- `chaseStuckTime: number` — time when stuck check started
- `harborWaitStart: number?` — if non-nil, NPC is waiting at Harbor boundary

### New client signal
- `NPCTeleported(npcId, fromPosition, toPosition)` — fired when NPC teleports
  due to stuck detection. Clients can show a poof/smoke VFX at both positions.

### Notes for next developer
- No Roblox Studio MCP server is configured for this project — couldn't test
  in-game. Changes are structurally verified via StyLua formatting.
- The stuck detection teleport picks the nearest waypoint in the NPC's zone.
  If the NPC has no patrol waypoints, it falls back to `spawnPosition`.
- Target re-evaluation during chase now resets `lastChaseRecalcTime = 0` when
  switching to a closer target, ensuring immediate path recomputation.

  NPC-005 (dormant mode), NPC-002 (skeleton loot pickup)

## 2026-03-02 — NPC-006: NPC Spawn Manager

### What was done
- Extended `src/server/Services/NPCService.lua` with a budget-driven spawn manager
  that replaces the old "one NPC per spawn point" approach.
- Added day/night budget scaling via `DayNightService.PhaseChanged` listener.
- Added Ghost Pirate budget tracking (stubbed until NPC-007 is implemented).
- Made respawning budget-aware: killed NPCs only respawn if under current budget.
- Added spawn point occupancy tracking to prevent double-spawning.

### Spawn budget system
1. **Day budget**: Random 6-10 Cursed Skeletons (from `GameConfig.CursedSkeleton`).
   Budget is rolled fresh each Dawn/Day transition.
2. **Night budget**: Day budget × 1.5 (rounded up). Example: day=8 → night=12.
   Ghost Pirate budget = random 4-6 (stubbed, NPC-007 not yet implemented).
3. **Dawn cleanup**: All Ghost Pirates despawn (drop carried doubloons), skeleton
   budget reduces to new day roll, excess skeletons trimmed (furthest from players).
4. **Budget-aware respawn**: Respawn queue checks `BudgetSkeletonCount < SkeletonBudget`
   before spawning. Over-budget respawns are silently skipped.
5. **Bonus NPC exclusion**: `SpawnBonusSkeleton()` and `SpawnAmbushNPCs()` mark NPCs
   as `isBonusNPC = true` and correct the budget counter so they don't count.

### Spawn point management
1. **Categorized spawn points**: `NPCSpawnPoints` children sorted into
   `SkeletonSpawnPoints` and `GhostPirateSpawnPoints` by `NPCType` attribute.
2. **Occupancy tracking**: `OccupiedSpawnPoints[part] = npcId` prevents two budget
   NPCs from sharing the same spawn point. Freed on despawn.
3. **Overflow spawning**: If all spawn points are occupied but budget has room,
   spawns at a random offset (10 studs) from an existing spawn point.
4. **Test spawn points**: If no `NPCSpawnPoints` exist in workspace, creates 10
   test points in a circle at radius 40 studs (development fallback).
5. **Ghost pirate point sharing**: If no dedicated ghost pirate spawn points exist,
   they share skeleton spawn points (ready for NPC-007).

### Budget trim logic
- `trimSkeletonBudget()` runs at Dawn to reduce skeleton count.
- Prefers despawning NPCs **furthest from any player** to minimize visual disruption.
- Iterates until `BudgetSkeletonCount <= SkeletonBudget`.

### Public API added
- `GetBudgetInfo()`: returns `{ skeletonBudget, skeletonActive, ghostPirateBudget, ghostPirateActive, bonusNPCs }`
- `HasBudgetRoom(npcType)`: returns whether spawn budget allows more of the given type

### Key design decisions
- **Phase-driven budget**: Budget is recalculated on each phase transition, not
  continuously adjusted. This gives predictable NPC populations per phase.
- **Randomized budgets**: Each day/night cycle rolls a new random count within the
  config range, adding variety between cycles.
- **Graceful trim**: Excess skeletons at Dawn are immediately despawned (furthest
  first), not left to die naturally. This ensures day budget is respected promptly.
- **Ghost Pirate stub**: All Ghost Pirate budget tracking and spawn point management
  is in place. When NPC-007 is implemented, it just needs to add a
  `spawnGhostPirate()` function and call it from `fillGhostPirateBudget()`.

### Notes for next developer
- Ghost Pirate spawning is stubbed — implement in NPC-007. The spawn manager already
  tracks `GhostPirateBudget`, `BudgetGhostPirateCount`, `GhostPirateNPCIds`, and
  `GhostPirateSpawnPoints`. Add `spawnGhostPirate()` and wire it into

5. **Rank change listener**: Uses `DataService.NotorietyRankChanged` server-side signal.

### CombatService changes (lunge + spin)
1. **Lunge (Rank 2)**: `LungeRequest` client signal → server validates rank, cooldown, state.
   0.3s windup delay → hit detection (8 stud range, 70° arc). Single-target like light/heavy.
   `LungeConfirm` fires to client for movement + VFX.
2. **Spin (Rank 4)**: `SpinRequest` client signal → server validates rank, cooldown, state.
   0.5s windup delay → multi-hit detection (6 stud range, 360° arc). Hits ALL targets in range.
   `SpinConfirm` fires to client for VFX.
3. **Both use `handlePlayerHitCustom()`** — shared hit logic with custom ragdoll/spill/knockback.
4. **Block speed fix**: CombatService now queries `RankEffectsService:GetWalkSpeed()` for
   the correct default speed when entering/exiting block (accounts for rank speed bonus).

### CombatController changes (client input)
1. **X key**: Lunge attack (gated by `HasLungeUnlock` from server)
   - Forward dash trail VFX (gold/orange), thrust animation, movement via LinearVelocity
2. **C key**: Spin attack (gated by `HasSpinUnlock` from server)
   - Circular particle burst VFX (gold/orange), fast spin animation
3. **State checks**: Both lunge and spin prevent other actions during execution
   (IsLunging/IsSpinning flags checked by canSwing, block, dash, primary attack)
4. **Ragdoll cleanup**: Lunge/spin state cancelled on ragdoll, character respawn

### Key bindings chosen
- **X**: Lunge — avoids conflict with F (ship lock ProximityPrompt)
- **C**: Spin — avoids conflict with E (ship deposit ProximityPrompt)
- No conflict with: WASD (move), Q (dash), G (shop), Mouse1 (swing), Mouse2 (block),
  E/F/R (ship ProximityPrompts)

### Design decisions
- **Per-player pickup radius**: DoubloonService now builds a `radiusSq` per player each
  collection check, rather than using a single global constant. Minimal perf impact since
  the check already iterates all players.
- **Multi-hit for spin**: Spin uses `performMultiHitDetection()` which returns arrays of
  all hit players/NPCs/containers. Each target gets independent ragdoll/spill/knockback.
  This makes spin powerful but fair — shorter range (6 studs) and long cooldown (5s).
- **Lunge dash is server-confirmed**: Server validates then fires `LungeConfirm` to client.
  Client applies LinearVelocity for the 6-stud forward dash. Hit detection happens
  server-side after the windup delay.

  the bonus. CombatService queries `GetWalkSpeed()` when applying block speed reduction
  to ensure the 50% block speed is applied to the rank-boosted speed, not the base.
- **Title re-attach**: BillboardGui is re-created on CharacterAdded (not persisted on the
  old character). PointLight glow is parented to Head for proper position.

## 2026-03-02 — DAYNIGHT-001: Day/Night Cycle Clock & Sync

### What was done
- Created `src/server/Services/DayNightService.lua` — server-authoritative day/night
  cycle clock that cycles through Dawn → Day → Dusk → Night on a Heartbeat loop.
- Created `src/client/Controllers/DayNightController.lua` — client-side sync controller
  that mirrors the server phase and exposes it to other controllers.

### DayNightService.lua — server
- **Clock**: Runs on `RunService.Heartbeat`, advancing `CycleElapsed` which wraps at
  `totalCycleDuration` (660s = 11 minutes). Uses modular arithmetic for seamless looping.
- **Phase derivation**: `getPhaseFromElapsed()` walks the ordered phase list
  (Dawn/Day/Dusk/Night) with precomputed durations from `GameConfig.DayNight`.
- **Phase transitions**: Detects when `CurrentPhase` changes and fires:
  - Server-side `PhaseChanged` signal (GoodSignal): `(newPhase, previousPhase)` — for
    other services (NPC spawning, threat, container rates, etc.)
  - Client `PhaseChanged` signal (Knit RemoteSignal): `FireAll(phase, 0, phaseDuration)`
    — broadcasts to all connected players simultaneously
- **Start phase**: Initializes at the beginning of the Day phase so players joining
  a fresh server start in daytime.
- **Server API**: `GetCurrentPhase()`, `IsNight()`, `IsDay()`, `IsDawn()`, `IsDusk()`,
  `GetPhaseProgress()`, `GetPhaseDuration()`, `GetPhaseElapsed()`, `GetCycleElapsed()`,
  `GetDurationForPhase(phase)`
- **Client-callable**: `Client:GetCurrentPhase()`, `Client:GetTimeInfo()` — GetTimeInfo
  returns `{ phase, phaseElapsed, phaseDuration, cycleElapsed, totalCycle }` for full
  initial sync

### DayNightController.lua — client
- **Initial sync**: On KnitStart, calls `DayNightService:GetTimeInfo()` to get the
  server's current phase and elapsed time (handles mid-cycle joins).
- **Phase tracking**: Listens to `DayNightService.PhaseChanged` signal and updates
  local `CurrentPhase`, `PhaseElapsed`, `PhaseDuration`.
- **Local signal**: `DayNightController.PhaseChanged` (GoodSignal) fires
  `(newPhase, previousPhase)` so other controllers (future HUD, lighting, banners)
  can react without needing a server reference.
- **Public API**: `GetCurrentPhase()`, `IsNight()`, `IsDay()`, `IsDawn()`, `IsDusk()`,
  `GetPhaseElapsed()`, `GetPhaseDuration()`, `GetPhaseProgress()`

### Design decisions
- Uses `Heartbeat` instead of `task.wait()` loop for frame-accurate timing and to
  avoid blocking the service thread

## 2026-03-02 — LOOT-005: Doubloon Scatter & Pickup System

### What was done
- Created `src/server/Services/DoubloonService.lua` — a server-authoritative Knit
  service that manages all doubloon pickup entities in the game world.

### DoubloonService.lua — complete implementation

#### Pickup entity creation
- Doubloon pickups are small gold cylinder Parts (coin shape) in a `workspace.Pickups`
  folder, with a PointLight for gold glow
- Each pickup stores its doubloon value as an attribute (`DoubloonValue`)
- Parts are Anchored, CanCollide=false, CanQuery=false, CanTouch=false, CastShadow=false
  for minimal physics overhead

#### Scatter system — `ScatterDoubloons(position, amount, scatterRadius)`
- Primary API entry point for other services (container breaks, PvP spills, NPC drops)
- Splits total doubloon amount into 3–15 individual pickup entities using intelligent
  splitting: small amounts get 1 pickup per doubloon, larger amounts split into
  baseValue + remainder distribution
- Each pickup lands at a random angle and distance within the scatter radius
- Pickups placed 0.5 studs above center Y for ground visibility

#### Proximity collection (server-authoritative)
- Heartbeat loop checks every 0.1 seconds (not every frame) for performance
- Builds player position list from HumanoidRootPart, then checks each pickup against
  each player using squared XZ distance (avoids sqrt)
- When player is within `GameConfig.Pickups.pickupRadius` (4 studs), the pickup is
  collected: value added to `heldDoubloons` via SessionStateService, part destroyed,
  client signal fired for VFX/SFX
- First-come-first-served: closest check wins when multiple players are near

#### Despawn system
- Every check cycle, expired pickups (>15s since creation) are removed
- Oldest-first removal ensures bounded memory

#### Global pickup cap
- Before spawning new pickups, `enforcePickupCap()` removes oldest pickups if the
  count would exceed `GameConfig.Pickups.maxLoosePickups` (200)
- Makes headroom for the incoming batch before creating new entities

#### Disconnect spill integration
- Connects to `DataService.DisconnectDoubloonSpill` signal in KnitStart
- When a player disconnects with held doubloons, scatters them at their last position
  with a 4-stud radius

#### Signals
- **Server-side**: `DoubloonService.DoubloonCollected` (GoodSignal) fires
  `(player, amount, position)` on each pickup collection — for future stat tracking,
  notoriety XP, etc.
- **Client-side**: `DoubloonService.Client.DoubloonCollected` (Knit RemoteSignal) fires
  `(pickupPosition, amount)` to the collecting player — for VFX/SFX feedback

### Public API
- `ScatterDoubloons(position, amount, scatterRadius?)` — scatter pickups at a position
- `GetActivePickupCount()` — returns current number of active pickups

## 2026-03-02 — COMBAT-005: Ragdoll Physics System

### What was done
- Created `src/shared/RagdollModule.lua` — a shared physics-based ragdoll module
  that replaces the old placeholder PlatformStanding ragdoll.
- Updated `src/client/Controllers/CombatController.lua` to use RagdollModule for
  all ragdoll visuals with knockback support.
- Updated `src/server/Services/CombatService.lua` to calculate and send knockback
  velocity to the client on player hits.
- Extended `src/shared/GameConfig.lua` with knockback force values per hit type.

### RagdollModule.lua — complete implementation

#### Joint disabling system
- Iterates all Motor6D joints in the character and disables them (Enabled = false)
- Creates BallSocketConstraint replacements at each joint position using the
  motor's C0/C1 CFrame transforms
- Creates Attachment pairs on Part0/Part1 for constraint anchoring
- All joints, constraints, and attachments are tracked for clean restoration

#### Per-joint limits (exaggerated for comedy)
- 14 named joints with individual angle limits:
  - Neck: 50° upper, ±35° twist (head bobble)
  - Waist: 35° upper, ±25° twist (torso bending)
  - Shoulders: 130° upper, ±90° twist (floppy arm comedy)
  - Elbows: 130° upper, ±10° twist (wide bend)
  - Hips: 100° upper, ±35° twist (exaggerated leg flailing)
  - Knees: 100° upper, ±5° twist (dramatic leg folding)
  - Wrists/Ankles: 50° upper, moderate twist
- Slight restitution (0.3) on all joints for bouncy tumble feel
- Joints not in the named table get default limits (90°, ±45° twist)

#### Limb collision system
- Enables CanCollide on 13 body parts (Head, UpperTorso, LowerTorso, all arm
  and leg segments) so ragdolled characters interact with terrain/walls
- Original CanCollide values stored and restored on ragdoll end
- HumanoidRootPart stays non-collidable to avoid physics interference

#### Knockback impulse
- Sets AssemblyLinearVelocity on HumanoidRootPart for immediate impulse
- Adds upward kick (35% of horizontal magnitude) for exaggerated launch
- Adds angular velocity perpendicular to knockback direction for comedic spin
  (capped at 15 rad/s to prevent excessive rotation)

#### Humanoid state management
- PlatformStanding = true to prevent auto-stand during ragdoll
- Disables GettingUp and Running states to prevent Humanoid interference
- On restore: re-enables states, sets PlatformStanding = false, triggers
  GettingUp state for smooth stand-up animation

### CombatController.lua changes
- **Removed**: Old `applyRagdollVisual()` function (PlatformStanding placeholder)
- **Added**: RagdollModule import and usage
- **Updated onRagdollTrigger**: Now accepts `knockbackVelocity: Vector3?` parameter,
  calls `RagdollModule.enable(character, knockbackVelocity)` instead of
  PlatformStanding, schedules `RagdollModule.disable()` after duration
- **Added RagdollCleanupThread**: Tracks the pending disable task so overlapping
  ragdolls cancel the old cleanup before starting a new one
- **Added CharacterRemoving handler**: Calls `RagdollModule.cleanup(character)` and
  resets all ragdoll state (IsRagdolled, RagdollCleanupThread) on respawn/death

### CombatService.lua changes
- **Updated RagdollTrigger signal**: Now sends `knockbackVelocity: Vector3` as the
  4th argument (was previously just attackerName + duration)
- **Added knockback calculation in handlePlayerHit**: Computes knockback direction
  from attacker→target XZ vector (normalized), multiplied by
  `GameConfig.Ragdoll.lightHitKnockback` (15 studs/s)
- **Fallback**: If attacker and target overlap, uses attacker's look direction
- **Fixed variable shadowing**: Renamed inner `targetHRP` to `spillHRP` in the
  loot spill section to avoid shadowing the outer variable

### GameConfig.lua changes
- **Extended RagdollConfig type**: Added `lightHitKnockback`, `heavyHitKnockback`,
  `blockedHitKnockback` number fields
- **Added knockback values**:
  - `lightHitKnockback = 15` — studs/s impulse for stumble
  - `heavyHitKnockback = 40` — studs/s impulse for launch (future COMBAT-002)
  - `blockedHitKnockback = 0` — no knockback when blocked (future COMBAT-003)

### Design decisions
- Module is in src/shared/ for potential future server-side NPC ragdoll use
- BallSocketConstraint chosen over HingeConstraint for more chaotic tumbling
  (matches the "exaggerated and satisfying" spec requirement)
- Knockback is computed server-side and sent to client so it can't be exploited
- Upward kick and angular velocity are computed from knockback magnitude, meaning
  stronger hits automatically produce more dramatic tumbles
- Joint limits are intentionally loose (especially shoulders/elbows) so arms flail
  dramatically during ragdoll — this is the "core humor element" from the spec

## 2026-03-02 — SHIP-001: Ship Dock & Spawn System

### What was done
- Created `src/server/Services/ShipService.lua` — server-authoritative ship

### ShipService.lua — complete system

#### Dock slot management
- Dock slots defined as Parts in `workspace.ShipDockPoints` with `SlotIndex`
  attribute (1-24).
- `getDockPoints()` fetches all dock point Parts, sorted by SlotIndex.
- `getAvailableDockSlot()` returns first unoccupied slot.
- `DockSlotOccupancy: { [number]: ShipEntry }` tracks slot → ship mapping.
- `DockedShips: { [Player]: ShipEntry }` tracks player → ship mapping.
- Max 24 simultaneous docked ships per the GameConfig.ShipSystem.maxDockSlots.

#### Ship spawning
- Triggers on `DataService.PlayerDataLoaded` signal (not PlayerAdded), because
  ship tier requires treasury balance from persistent data.
- `spawnShip(player)` calculates tier via `GameConfig.getShipTierForTreasury()`,
  finds available dock slot, creates placeholder model, registers in both
  DockedShips and DockSlotOccupancy registries.
- Handles studio edge case (players already in game when Knit starts).

#### Ship model creation (placeholder)
- 7 ship tier appearances defined in `SHIP_APPEARANCE` table (rowboat through
  ghost_ship) with size, color, and material per tier.
- Hull part with metadata attributes: ShipSlotIndex, ShipTierId, ShipTierName,
  OwnerName.
- Mast part for tiers >= 2 (Sloop and above).
- Ghost Ship (tier 7) gets spectral PointLight and ParticleEmitter glow.
- BillboardGui with owner name and ship tier name label (visible from 60 studs).
- Ships parent to `workspace.DockedShips` folder.

#### Ship despawning
- On PlayerRemoving: removes from DockedShips but keeps in DockSlotOccupancy,
  starts 30-second delayed despawn via `task.delay`.
- `PendingDespawns: { [number]: thread }` tracks userId → despawn thread.
- If player rejoins before timer expires, pending despawn is cancelled and old
  ship is cleaned up so a fresh one spawns.
- `BindToClose` immediately destroys all ship models and cancels pending
  despawns on server shutdown.

#### Ship tier recalculation
- `RecalculateShipTier(player)` checks treasury against
  `GameConfig.getShipTierForTreasury()`, and if tier changed: destroys old model,
  creates new model at same dock position, fires tier change signals.
- Called by future SHIP-002/003 when treasury changes (deposit+lock, gear
  purchase, cosmetic purchase).

#### Server-side API
- `GetDockedShip(player)` — full ShipEntry for a player
- `GetShipAtDock(slotIndex)` — ShipEntry at a dock slot
- `GetShipOwnerUserId(slotIndex)` — owner UserId at a slot
- `GetShipModel(player)` — the workspace Model
- `GetShipPosition(player)` — dock position Vector3
- `GetDockSlot(player)` — dock slot number
- `GetAllDockedShips()` — summary array of all docked ships

#### Client-callable methods (Knit RPCs)
- `Client:GetMyShip(player)` → slotIndex, shipTierId, position
- `Client:GetAllDockedShips(player)` → array of ship summaries

#### Client signals (Knit RemoteSignals)
- `ShipSpawned` — FireAll(slotIndex, ownerUserId, ownerName, shipTierId, position)
- `ShipDespawned` — FireAll(slotIndex, ownerUserId)
- `ShipTierChanged` — FireAll(slotIndex, ownerUserId, newShipTierId)

#### Server-side signals (GoodSignal)
- `ShipSpawned` — (shipEntry)
- `ShipDespawned` — (shipEntry)
- `ShipTierChanged` — (shipEntry, oldTierId, newTierId)

## 2026-03-02 — THREAT-001: Threat Level Tracking & Accumulation

### What was done
- Created `src/server/Services/ThreatService.lua` — server-authoritative threat
  accumulation service that drives the PvPvE tension mechanic.

### ThreatService.lua — complete implementation
- **Time-based accumulation**: +5 per minute since last lock, computed per-second
  from `GameConfig.Threat.timeRate / 60`. Uses per-player `TimeAccumulators` that
  accumulate delta-time and fire every 1 second.
- **Held doubloons check**: Every 30 seconds (configurable via
  `GameConfig.Threat.heldDoubloonsInterval`), calculates
  `floor(heldDoubloons / 50) * 1` threat. Per-player `HeldDoubloonTimers` track
  the interval.
- **Container break hook**: Subscribes to `ContainerService.ContainerBroken` signal.
  Adds +2 threat (from `GameConfig.Threat.containerBreakGain`) to the breaking
  player.
- **NPC kill hook**: `OnNPCKilled(player)` method exposed for future NPC services.
  Adds +3 threat (from `GameConfig.Threat.npcKillGain`).
- **Danger zone check**: `IsInDangerZone(player)` returns false (placeholder until
  ZONE-001). When implemented, +3/min in danger zones via `DangerZoneAccumulators`.
- **Night multiplier**: All threat gains multiplied by 1.5x during Night phase
  via `getNightMultiplier()` using `DayNightService:IsNight()`.
- **Harbor pause**: `IsInHarbor(player)` checks for a workspace `HarborZone` Part
  using AABB bounding box detection. All periodic and event-driven threat gains
  skip players inside the Harbor. When HARBOR-001 places a `HarborZone` part in
  workspace, the pause activates automatically.
- **Central `AddThreatWithMultiplier` method**: All threat sources route through
  this, which applies night multiplier and harbor check before delegating to
  `SessionStateService:AddThreat()`.

### Integration with existing services
- **ContainerService**: Already fires `ContainerBroken(containerEntry, attackingPlayer)`
  signal — ThreatService subscribes to it. No changes to ContainerService needed.
- **ShipService**: Already handles threat reduction:
  - `DepositAll`: calls `SessionStateService:AddThreat(player, -25)`
  - `LockShip`: calls `SessionStateService:SetThreatLevel(player, 0)`
  No changes to ShipService needed.
- **SessionStateService**: Already has full threat API (`AddThreat`, `SetThreatLevel`,
  `GetThreatLevel`, `ResetThreat`, `GetLastLockTime`) and replicates `threatLevel`
  to the client via `SessionStateChanged` signal. No changes needed.
- **DayNightService**: Used for night multiplier check. No changes needed.
- **Auto-discovery**: Server `Main.server.lua` auto-loads all services in the
  Services folder. ThreatService is picked up automatically.

### Cleanup
- Per-player timer tables (`HeldDoubloonTimers`, `TimeAccumulators`,
  `DangerZoneAccumulators`) are cleaned up on `Players.PlayerRemoving`.
- Heartbeat tick runs every 1 second to keep CPU overhead minimal while providing
  responsive threat updates.

## 2026-03-02 — NPC-003: NPC SimplePath Patrol Behavior

### What was done
- Created `src/shared/SimplePath.lua` — lightweight PathfindingService wrapper module
  that provides a clean API for NPC navigation with signals (Reached, Blocked, Error).
- Modified `src/server/Services/NPCService.lua` to use SimplePath for NPC patrol
  instead of raw `Humanoid:MoveTo()` with random positions.
- Added `HarborService:IsPositionInHarbor()` public method to `src/server/Services/HarborService.lua`
  for NPC waypoint filtering.

### SimplePath module (src/shared/SimplePath.lua)
1. **Wraps PathfindingService**: Creates paths with configurable agent params
   (AgentRadius, AgentHeight, AgentCanJump).
2. **API**: `SimplePath.new(agent, agentParams)`, `:Run(destination)`, `:Stop()`,
   `:IsRunning()`, `:Destroy()`.
3. **Signal-based**: `Reached` fires when destination reached, `Blocked` fires when
   path obstructed mid-traversal, `Error` fires on path computation failure.
4. **MoveToFinished-driven**: Follows path waypoints using `Humanoid:MoveTo()` with
   automatic progression via `MoveToFinished`. Handles jump waypoints.
5. **Heartbeat-compatible**: `setState()` in NPCService calls `:Stop()` when leaving
   patrol state, preventing conflicts with chase `Humanoid:MoveTo()` calls.

### Patrol waypoint system
1. **PatrolWaypoints folder**: Reads `workspace.PatrolWaypoints` for patrol waypoint Parts.
   Each Part has `Zone` (string) and optional `Order` (number) attributes.
   Waypoints are sorted by Order within each zone.
2. **Fallback to spawn points**: If no PatrolWaypoints folder exists, uses NPC spawn
   points of the same zone as patrol destinations (requires 2+ points per zone).
3. **Generated fallback**: If a zone has <2 waypoints, generates 4 positions in a
   circle (15-stud radius) around the NPC's spawn point.
4. **Harbor exclusion**: All waypoints inside the Harbor safe zone are filtered out
   during discovery (uses `HarborService:IsPositionInHarbor()`).

### NPCEntry changes
- Added `simplePath` (SimplePath instance), `patrolWaypoints` (Vector3 list),
  `patrolWaypointIndex` (current position in loop) to NPCEntry type.
- SimplePath created per NPC at spawn time with agent params from `GameConfig.NPCBehavior`.
- Each NPC starts at a random waypoint index to avoid clustering.
- SimplePath signals wired at spawn: Reached sets `lastPatrolMoveTime` for pause,
  Blocked/Error skip pause to try next waypoint immediately.

### AI state machine changes
1. **PATROL state**: Checks `simplePath:IsRunning()`. If not running, waits 2 seconds
   at waypoint, then advances to next waypoint and calls `simplePath:Run()`.
   Falls back to random wander for NPCs without SimplePath (shouldn't happen in practice).
2. **setState()**: Calls `simplePath:Stop()` when transitioning out of PATROL state
   (e.g., to CHASE). This disconnects MoveToFinished so chase `Humanoid:MoveTo()`
   calls don't conflict.
3. **despawnNPC()**: Calls `simplePath:Destroy()` to clean up connections.
4. **JumpPower**: Changed from 0 to 50 to enable pathfinding jump waypoints.

### Key design decisions
- **SimplePath in shared/**: Module is in `src/shared/` so it can be reused when
  NPC-004 upgrades chase behavior to use SimplePath as well.
- **Signal-based integration**: SimplePath uses GoodSignal events rather than coroutines
  or task.spawn, keeping the API clean and compatible with the heartbeat AI loop.
- **Patrol pause at waypoints**: 2-second pause at each waypoint makes NPCs look like
  they're "surveying" their zone rather than robotically walking between points.
- **Random start index**: Each NPC starts at a random patrol waypoint so multiple
  NPCs in the same zone don't all walk the same route in lockstep.
- **Blocked = skip**: When SimplePath encounters a blocked path, the NPC skips to the
  next waypoint rather than retrying. This prevents NPCs from getting stuck on one
  unreachable point.
- **Chase still uses MoveTo**: NPC-004 will upgrade chase behavior to use SimplePath.
  For now, chase uses `Humanoid:MoveTo()` directly (same as before).

### Notes for next developer
- NPC-004 (SimplePath Chase): The SimplePath module is already available. In chase state,
  replace `entry.humanoid:MoveTo(targetPos)` with `entry.simplePath:Run(targetPos)`
  on a 0.3s recalculation interval. Add stuck detection (3s recalc, 6s teleport).
- NPC-005 (Dormant Mode): When dormant, call `entry.simplePath:Stop()` and skip the
  AI update entirely. Resume patrol when player approaches <150 studs.
- PatrolWaypoints folder: Map builders should create Parts in `workspace.PatrolWaypoints`
  with `Zone` attribute matching spawn point zones. Use `Order` attribute (1, 2, 3...)
  to define patrol route. Without this, NPCs patrol between spawn points.
- The `getPatrolWaypointsForZone()` fallback generates positions at Y=0 offset from
  spawn. If terrain is uneven, the generated positions may not be reachable — define
  explicit PatrolWaypoints for hilly zones.
- Next priorities: NPC-004 (SimplePath chase), NPC-007 (Ghost Pirate), NPC-008
  (Phantom Captain), NPC-005 (dormant mode)

## 2026-03-02 — NPC-005: NPC Dormant Mode & Performance Budget

### What was done
- Extended `src/server/Services/NPCService.lua` with three performance optimizations:
  dormant mode, per-frame path calculation budget, and staggered NPC AI updates.

### Dormant mode
1. **Distance check**: Each Heartbeat, a cached list of player HumanoidRootParts is
   built once (`rebuildPlayerHRPCache`). Each NPC's squared distance to the nearest
   player is computed via `nearestPlayerDistSq` (avoids sqrt per NPC).
2. **Enter dormant**: If an NPC is >150 studs from all players, `enterDormant()` is
   called: SimplePath is stopped, WalkSpeed set to 0, `isDormant = true`.
3. **Exit dormant**: When a player comes within 150 studs, `exitDormant()` restores
   base WalkSpeed (effective speed recalculated by `updateNPCAI` on next tick) and
   resets patrol timing so the NPC resumes smoothly.
4. **Dormant NPCs skip AI**: The Heartbeat loop `continue`s past dormant NPCs —
   no state machine, no aggro checks, no pathfinding.
5. **Wake on damage**: `DamageNPC()` calls `exitDormant()` before processing damage,
   so a player hitting an NPC at the boundary edge wakes it immediately.

### Path calculation budget
1. **`budgetedPathRun(entry, target)`**: Wraps all `SimplePath:Run()` calls. If the
   per-frame budget (`maxPathRecalcsPerFrame = 3`) is not exhausted, the path starts
   immediately and the counter increments. Otherwise, the request is queued.
2. **`PathQueue`**: Array of `{ entry, target }` for deferred path requests.
3. **`drainPathQueue()`**: Called at the start of each Heartbeat, processes queued
   requests up to the frame budget. Dead/dormant entries are silently dropped.
4. **All 4 SimplePath:Run() call sites** in `updateNPCAI` (patrol waypoint advance,
   loot pickup path, leash-return path, chase recalc) now go through `budgetedPathRun`.

### Staggered NPC updates
1. **2-frame round-robin**: Each NPC ticks based on `entry.id % NPC_UPDATE_STAGGER`
   vs `FrameCounter % NPC_UPDATE_STAGGER`. Non-urgent NPCs (idle, patrol, loot_pickup)
   only update every 2nd frame.
2. **Urgent states bypass stagger**: NPCs in CHASE, ATTACK_SLASH, ATTACK_LUNGE, or
   FLINCH always update every frame for responsive combat.
3. **FrameCounter**: Incremented each Heartbeat, used for stagger scheduling.

### NPCEntry type changes
- Added `isDormant: boolean` field (defaults to `false` in both `spawnSkeleton` and
  `spawnGhostPirate`).

### GameConfig values used (already existed)
- `NPCBehavior.dormantDistance = 150` studs
- `NPCBehavior.maxPathRecalcsPerFrame = 3`

### Performance characteristics
- **Player HRP cache**: O(P) rebuild per frame (P = player count), O(P) per NPC
  distance check. Total dormant check: O(N×P) per frame.
- **Squared distance**: No sqrt calls — cheaper comparison for dormant threshold.
- **Path budget**: Guarantees ≤3 PathfindingService computations per frame regardless
  of NPC count. Queued paths resolve within 1-2 frames under normal load.
- **Stagger**: Halves per-frame AI tick count for non-combat NPCs. Patrol and loot
  pickup behavior is time-based (not frame-based), so 2-frame stagger is invisible
  to players.

### Notes for next developer
- The stagger constant `NPC_UPDATE_STAGGER = 2` can be increased (e.g., 3) if NPC
  count grows significantly. Higher values save more CPU but add latency to patrol
  waypoint transitions.
- Dormant NPCs still exist in `ActiveNPCs` and their models are replicated to clients.
  A future optimization could hide dormant NPC models on clients far from them.
- The path queue has no priority system — all queued paths are first-come-first-served.
  Chase paths are naturally prioritized because chase NPCs update every frame and
  request paths more frequently.
- Next priorities: NPC-008 (Phantom Captain), NPC-009 (night NPC modifiers),
  LOOT-006 (player loot visibility)

   for ragdoll + launch), VentEruptionLaunchVFX (all clients, for spectator effects).
6. **Public API**: GetVentStates() (server+client RPC for late-join sync),
   IsPositionOnVent(position) for other services.

1. **Dormant VFX**: Faint gray steam particles (Rate=3, slow upward drift).
2. **Warning VFX**: Steam rate increases to 8, orange-tinted heavy steam (Rate=12),
   PointLight orange glow tweens from 0→3 brightness over 4.5s, looped rumble Sound
   fades in over 3s with 3D rolloff (10-80 studs).
3. **Eruption VFX**: Fire geyser particle column (yellow→red gradient, Speed=30-50,
   Rate=40, LightEmission=1), bright red-orange PointLight (Brightness=5, Range=40),
   eruption burst Sound with 3D rolloff (15-120 studs).
4. **Eruption hit**: Shows "Volcanic eruption!" notification, enables RagdollModule with
   launch velocity, schedules cleanup after ragdoll duration.
5. **Phase cleanup**: Each phase transition destroys previous phase's emitters/lights/sounds.
   Dormant steam emitter persists across phases (rate adjusted: 3→8→0→3).
6. **Late-join sync**: Calls GetVentStates RPC on start, applies current phase VFX.
7. **Character cleanup**: CharacterRemoving cleans up ragdoll state.

### Design decisions

- **Staggered initial timers**: Zones start with 15-45s random idle timers so they don't all
  surge at once on server start.

## 2026-03-02 — UI-005: HUD Minimap

### What was done
- Created `src/client/UI/MinimapPanel.lua` — Fusion 0.3 component that renders a circular
  minimap frame in the bottom-left corner of the screen. Contains a player direction arrow
  (triangle pointing up = player's forward direction), a subtle 75% range ring for distance
  reference, and a dots container where MinimapController adds entity markers. Uses UITheme
  design tokens, circular UICorner, and ClipsDescendants for visual containment. 160x160 pixel
  map with semi-transparent dark background and stroke border.
- Created `src/client/Controllers/MinimapController.lua` — Knit controller that manages all
  minimap entity tracking and rendering. Creates its own ScreenGui (DisplayOrder 15). M keybind
  toggles visibility. Heartbeat update loop throttled to 0.1s (10 updates/second) for
  performance. Tracks entity dots via a pooling system (entityId → instance map) with automatic
  stale dot cleanup each frame.

### Entity rendering
- **Other players**: White 6px circle dots. Hidden when outside 150-stud range.
- **Bounty target**: Gold skull icon (💀), 12px. Clamped to circle edge when out of range
  (with 0.3 transparency to indicate distance). Overrides normal player dot.
- **Skeletons**: Red 6px circle dots. Hidden when outside range.
- **Ghost Pirates**: Faint red 5px dots, only shown within 20 studs. Sine-based flicker
  (~60% visibility, 8 Hz oscillation) with 0.3 transparency. Invisible beyond 20 studs.
- **Phantom Captains**: Purple 8px circle dots. Standard range visibility.
- **Own ship**: Cyan anchor icon (⚓), 12px. Clamped to edge when out of range.
- **Active events**: Gold star icon (⭐), 12px. Clamped to edge when out of range.

### Coordinate system
Uses CFrame:PointToObjectSpace for world-to-minimap transform. The minimap rotates with
the player's facing direction: "up" on the minimap always means "forward" for the player.
Object space X maps to minimap right, object space Z maps to minimap down (since positive Z
in object space is "behind" the player). Distance-based circular culling (not rectangular)
ensures dots only appear within the visible circle.

### Edge clamping
Important entities (bounty target, own ship, active events) use worldToMinimapClamped()
which normalizes the direction vector to the circle edge when out of range, with reduced
opacity (0.3 TextTransparency) to indicate the entity is beyond the visible range.

### Acceptance criteria met
- Minimap visible in screen corner (bottom-left, 160x160) ✓
- Players shown as white dots ✓
- NPCs shown as red dots ✓
- Ghost Pirates only flicker within 20 studs ✓
- Bounty target shown as skull ✓
- Events and own ship marked with distinct icons (star, anchor) ✓
- Minimap updates in real-time (10 Hz Heartbeat loop) ✓
- Scale and rotate relative to player facing direction ✓

### Design decisions
- **Standalone controller pattern**: Like LeaderboardController and PlayerStatsController,
  the minimap uses its own ScreenGui and is fully self-contained. No HudController wiring
  needed, which keeps the HUD clean.
- **Instance.new for dots**: Entity dots are managed with Instance.new (not Fusion) because
  they're highly dynamic — created, destroyed, and repositioned every 0.1s. Fusion's
  reactive system adds overhead for elements that need imperative position updates.
- **Dot pooling with stale cleanup**: Each update increments a counter. Dots mark themselves
  as "seen" via getOrCreateDot(). After all entities are processed, cleanupStaleDots()
  destroys any dot not seen this frame. This handles entity spawning, despawning, and type
  transitions (e.g., player gaining/losing bounty) automatically.
- **NPC type from model name prefix**: Instead of reading attributes from child Parts,
  the NPC type is determined by the model name prefix (Skeleton_, GhostPirate_,
  PhantomCaptain_) which is simpler and avoids iterating child parts.
- **No north indicator**: In a rotating minimap, a fixed "N" label would need per-frame
  rotation positioning. Omitted for simplicity — the player arrow and relative entity
  positions provide sufficient spatial awareness.
- **M keybind**: Intuitive ("M for Map"), unused by any other system
  (G=shop, L=leaderboard, P=stats, Q=dash, X=lunge, C=spin, E/F/R=ship).

### Notes for next developer
- MAP_RANGE (150 studs) controls how much of the world is visible on the minimap.
  Adjust in MinimapController.lua if the game map is larger/smaller than expected.
- UPDATE_INTERVAL (0.1s) can be increased for better performance on low-end devices,
  or decreased for smoother dot movement at the cost of CPU.
- The minimap uses ClipsDescendants which clips rectangularly (Roblox limitation), but
  circular culling in worldToMinimap prevents dots from appearing in the "corners" outside
  the visual circle.
- Ghost pirate flicker uses math.sin(tick() * 8) > -0.2 for a natural, irregular flicker
  when sampled at the 10 Hz update rate.
