## 2026-03-02 — DAYNIGHT-001: Day/Night Cycle Clock & Sync

### What was done
- Created `src/server/Services/DayNightService.lua` — server-authoritative day/night
  cycle clock that cycles through Dawn → Day → Dusk → Night on a Heartbeat loop.
- Created `src/client/Controllers/DayNightController.lua` — client-side sync controller
  that mirrors the server phase and exposes it to other controllers.

### DayNightService.lua — server
- **Clock**: Runs on `RunService.Heartbeat`, advancing `CycleElapsed` which wraps at
  `totalCycleDuration` (660s = 11 minutes). Uses modular arithmetic for seamless looping.
- **Phase derivation**: `getPhaseFromElapsed()` walks the ordered phase list
  (Dawn/Day/Dusk/Night) with precomputed durations from `GameConfig.DayNight`.
- **Phase transitions**: Detects when `CurrentPhase` changes and fires:
  - Server-side `PhaseChanged` signal (GoodSignal): `(newPhase, previousPhase)` — for
    other services (NPC spawning, threat, container rates, etc.)
  - Client `PhaseChanged` signal (Knit RemoteSignal): `FireAll(phase, 0, phaseDuration)`
    — broadcasts to all connected players simultaneously
- **Start phase**: Initializes at the beginning of the Day phase so players joining
  a fresh server start in daytime.
- **Server API**: `GetCurrentPhase()`, `IsNight()`, `IsDay()`, `IsDawn()`, `IsDusk()`,
  `GetPhaseProgress()`, `GetPhaseDuration()`, `GetPhaseElapsed()`, `GetCycleElapsed()`,
  `GetDurationForPhase(phase)`
- **Client-callable**: `Client:GetCurrentPhase()`, `Client:GetTimeInfo()` — GetTimeInfo
  returns `{ phase, phaseElapsed, phaseDuration, cycleElapsed, totalCycle }` for full
  initial sync

### DayNightController.lua — client
- **Initial sync**: On KnitStart, calls `DayNightService:GetTimeInfo()` to get the
  server's current phase and elapsed time (handles mid-cycle joins).
- **Phase tracking**: Listens to `DayNightService.PhaseChanged` signal and updates
  local `CurrentPhase`, `PhaseElapsed`, `PhaseDuration`.
- **Local signal**: `DayNightController.PhaseChanged` (GoodSignal) fires
  `(newPhase, previousPhase)` so other controllers (future HUD, lighting, banners)
  can react without needing a server reference.
- **Public API**: `GetCurrentPhase()`, `IsNight()`, `IsDay()`, `IsDawn()`, `IsDusk()`,
  `GetPhaseElapsed()`, `GetPhaseDuration()`, `GetPhaseProgress()`

### Design decisions
- Uses `Heartbeat` instead of `task.wait()` loop for frame-accurate timing and to
  avoid blocking the service thread
- Phase is derived from a single `CycleElapsed` counter rather than separate timers,
  making the system drift-free and easy to query
- Precomputes `PHASE_START` lookup and `PHASE_DURATIONS` from GameConfig at require-time
- Client receives elapsed-in-phase on sync so it can show accurate progress bars
- Starts at Day (not Dawn) because joining during dawn would be confusing for 30s

### Acceptance criteria verification
- [x] Server clock cycles through all 4 phases with correct durations (Dawn 30s,
  Day 360s, Dusk 30s, Night 240s = 660s total)
- [x] All players see the same time of day (server-synced via FireAll + GetTimeInfo)
- [x] Phase transition events fire for other systems (server GoodSignal + client signal)
- [x] Total cycle is 11 minutes (660 seconds)

### Notes for next developer
- Roblox Studio MCP server was not available for in-game testing; verified via
  StyLua formatting check across all src files
- Selene linting skipped due to pre-existing rokit PATH issue
- The service does NOT handle lighting/atmosphere changes — that's DAYNIGHT-002
- The service does NOT handle transition UI banners or audio — that's DAYNIGHT-003
- To consume phase changes from another service:
  `DayNightService.PhaseChanged:Connect(function(newPhase, prevPhase) ... end)`
- To consume from a controller:
  `DayNightController.PhaseChanged:Connect(function(newPhase, prevPhase) ... end)`
- Next highest-priority features: DAYNIGHT-002 (lighting/atmosphere), DAYNIGHT-003
  (transition banners/audio), DATA-002 (auto-save), LOOT-001 (container config —
  already in GameConfig, may just need marking done)

## 2026-03-02 — DATA-002: Auto-Save & Disconnect Grace-Save

### What was done
- Updated `src/server/Services/DataService.lua` with auto-save loop, disconnect
  grace-save, and server shutdown handling.

### Auto-save loop
- Runs every 60 seconds (configurable via `GameConfig.AutoSave.interval`)
- Calls `Profile:Save()` on all active profiles
- ProfileService handles DataStore retry logic and exponential backoff internally
- `SaveAllProfiles()` method exposed for other services if manual save is needed
- Logs count of profiles saved each cycle

### Disconnect grace-save
- `graceSaveOnDisconnect(player, profile)` runs inside `releaseProfile()` before
  `Profile:Release()` is called
- **Ship hold → treasury**: If `shipHold > 0`, adds the full amount to
  `profile.Data.treasury` so it persists. This means players don't lose banked
  doubloons on disconnect.
- **Held doubloons spill**: If `heldDoubloons > 0`, fires
  `DataService.DisconnectDoubloonSpill` signal with `(player, amount, position)`.
  Future LOOT-005 (pickup entity system) will connect to this signal to scatter
  doubloon pickups at the player's last position.
- Position is read from `HumanoidRootPart.Position` if the character still exists,
  otherwise falls back to `Vector3.new(0, 5, 0)`.

### Server shutdown handling
- `game:BindToClose()` releases all active profiles on server close
- This ensures data is saved even during server shutdowns/restarts
- Each profile goes through the full `releaseProfile()` flow including grace-save

### New signals
- `DataService.DisconnectDoubloonSpill` (GoodSignal): Fires when a disconnecting
  player has held doubloons that should be spilled. Args: `(player, amount, position)`.
  Currently no listeners — LOOT-005 will connect to this.

### Service dependency: SessionStateService
- DataService now gets a reference to SessionStateService in `KnitStart()` via
  `Knit.GetService("SessionStateService")`
- Grace-save reads `shipHold` and `heldDoubloons` from the session state before
  the session is cleaned up
- **Ordering**: DataService's `PlayerRemoving` handler connects before
  SessionStateService's (alphabetical load order: D < S), so session state is
  still available during grace-save. This is documented with a comment in the code.

## 2026-03-02 — LOOT-005: Doubloon Scatter & Pickup System

### What was done
- Created `src/server/Services/DoubloonService.lua` — a server-authoritative Knit
  service that manages all doubloon pickup entities in the game world.

### DoubloonService.lua — complete implementation

#### Pickup entity creation
- Doubloon pickups are small gold cylinder Parts (coin shape) in a `workspace.Pickups`
  folder, with a PointLight for gold glow
- Each pickup stores its doubloon value as an attribute (`DoubloonValue`)
- Parts are Anchored, CanCollide=false, CanQuery=false, CanTouch=false, CastShadow=false
  for minimal physics overhead

#### Scatter system — `ScatterDoubloons(position, amount, scatterRadius)`
- Primary API entry point for other services (container breaks, PvP spills, NPC drops)
- Splits total doubloon amount into 3–15 individual pickup entities using intelligent
  splitting: small amounts get 1 pickup per doubloon, larger amounts split into
  baseValue + remainder distribution
- Each pickup lands at a random angle and distance within the scatter radius
- Pickups placed 0.5 studs above center Y for ground visibility

#### Proximity collection (server-authoritative)
- Heartbeat loop checks every 0.1 seconds (not every frame) for performance
- Builds player position list from HumanoidRootPart, then checks each pickup against
  each player using squared XZ distance (avoids sqrt)
- When player is within `GameConfig.Pickups.pickupRadius` (4 studs), the pickup is
  collected: value added to `heldDoubloons` via SessionStateService, part destroyed,
  client signal fired for VFX/SFX
- First-come-first-served: closest check wins when multiple players are near

- When a player disconnects with held doubloons, scatters them at their last position

- **Client-side**: `DoubloonService.Client.DoubloonCollected` (Knit RemoteSignal) fires

## 2026-03-02 — LOOT-002: Container Spawning System

### What was done
- Created `src/server/Services/ContainerService.lua` — a server-authoritative Knit
  service that manages container spawning, tracking, damage, and breaking.

### ContainerService.lua — complete implementation

#### Spawn point system
- Containers spawn at pre-placed Parts in `workspace.ContainerSpawnPoints`
- Each spawn point Part can have a `Zone` attribute (string) for zone-based filtering
  (e.g., "beach", "jungle", "danger" — Cursed Chests only spawn at "danger" points)
- Occupied spawn points are tracked and never double-spawned
- Folder is auto-created with a warning if not found (map builders need to add Parts)

#### Weighted random container type selection
- Spawn frequency strings from GameConfig map to numeric weights:
  very_common=40, common=25, uncommon=15, rare=10, very_rare=5
- Night-only containers (cursed_chest) are excluded from the normal spawn pool
- Weighted selection table built at require-time for O(n) selection

#### Rolling timer spawn system
- Uses `RunService.Heartbeat` with an 8-second base spawn interval
- At night, interval is divided by `nightSpawnRateMultiplier` (1.5x → ~5.3s)
- Each tick attempts to spawn one container if below the 20-container cap
- 5 containers pre-spawned on server start so the map isn't empty

#### Night cycle integration
- Listens to `DayNightService.PhaseChanged` signal
- On Night start: spawns 2–3 Cursed Chests at random danger zone spawn points
  (shuffled selection, respects 20-container cap)
- On Dawn: despawns all remaining un-broken Cursed Chests (no loot drop)
- Containers spawned during night are flagged (`isNightSpawn = true`) for
  the 2x yield multiplier applied at break time

#### Container model creation (placeholder)
- Each container type has a distinct colored box model with appropriate size
- Cursed Chest: purple PointLight (range=30, visible through fog) + purple
  ParticleEmitter for eerie effect
- Captain's Vault: gold PointLight glow
- Container body Part stores attributes: ContainerId, ContainerType, MaxHP, CurrentHP
- Models placed under `workspace.Containers` folder

#### Damage & break system (for LOOT-003 integration)
- `DamageContainer(containerId, damage, player)` — deals damage, returns
  (destroyed: boolean, hpFraction: number?)
- `BreakContainer(containerId, player)` — calculates yield (with 2x night multiplier),
  calls `DoubloonService:ScatterDoubloons()`, fires client/server signals
- `RemoveContainer(containerId)` — frees spawn point, destroys model, updates count
- `GetContainerByPart(part)` — looks up container from a hit Part via ContainerId
  attribute (useful for raycast-based hit detection in COMBAT-001)

#### Client signals
- `ContainerSpawned` — fires to all players with (containerId, containerType, position)
- `ContainerDamaged` — fires to all players with (containerId, hpFraction) for VFX states
- `ContainerBroken` — fires to all players with (containerId, containerType, position)

#### Server signals
- `ContainerService.ContainerSpawned` — fires (containerEntry) for inter-service use
- `ContainerService.ContainerBroken` — fires (containerEntry, player) for stat tracking,
  threat accumulation, notoriety XP, etc.

#### Public API
- `DamageContainer(containerId, damage, player)` — deal damage to a container
- `BreakContainer(containerId, player)` — force-break a container
- `RemoveContainer(containerId)` — silently remove a container
- `GetContainer(containerId)` — get container entry by ID
- `GetContainerByPart(part)` — find container from a workspace Part
- `GetActiveContainerCount()` — current active count
- `SpawnContainerAt(containerTypeId, position)` — manual spawn for events/testing

## 2026-03-02 — COMBAT-001: Light Swing Attack

### What was done
- Created `src/server/Services/CombatService.lua` — a server-authoritative Knit
  service handling light swing combat (validation, hit detection, damage, ragdoll, spill).
- Created `src/client/Controllers/CombatController.lua` — client-side input handler
  for primary click attacks with swing animation, SFX, and ragdoll visual feedback.
- Extended `src/client/Controllers/SoundController.lua` with combat sound effects
  (swing swoosh, player hit, container hit, miss).

### CombatService.lua — complete implementation

#### Attack validation
- Checks session initialized, not ragdolling, not in recovery
- Server-side rate limiting: enforces 0.4s minimum between attacks (GameConfig.Combat.lightSwingCooldown)
- Per-player `LastAttackTime` tracking with cleanup on PlayerRemoving

#### Server-side hit detection
- `isInSwingArc(attackerCFrame, targetPosition)` — checks distance (8 studs)
  and angle (70° half-angle forward cone) from HumanoidRootPart CFrame
- Players checked first (PvP priority), then containers
- Per-target cooldown enforced (2s via SessionStateService:CanHitTarget)
- Ragdolled targets cannot be hit again (prevents chain-stun)
- Closest valid target in arc is selected

#### Player hit handling
- Records hit for per-target cooldown (SessionStateService:RecordHitTarget)
- Applies ragdoll (1.5s light hit via SessionStateService:StartRagdoll)
- Fires RagdollTrigger client signal to victim for visual feedback
- Calculates loot spill using GameConfig.calculateSpill (10%, bounty 2x)
- Deducts spilled doubloons from target's heldDoubloons
- Scatters doubloons at target position via DoubloonService
- Fires LootSpillVFX to all clients and PlayerHitPlayer server signal

#### Container hit handling
- Gets equipped gear damage from DataService → GameConfig.GearById
- Calls ContainerService:DamageContainer(containerId, damage, player)
- ContainerService already handles break, doubloon scatter, and signals
- Fires PlayerHitContainer server signal for stat tracking

#### Client signals
- `AttackRequest` — client fires to request a light swing
- `SwingResult` — server fires to attacker (hitType, targetName?)
- `RagdollTrigger` — server fires to victim (attackerName, ragdollDuration)
- `LootSpillVFX` — server fires to all (position, spillAmount)

#### Server signals
- `CombatService.PlayerHitPlayer` — (attacker, target, spillAmount)
- `CombatService.PlayerHitContainer` — (attacker, containerId, destroyed)

### CombatController.lua — complete implementation

- Respects `gameProcessed` to ignore clicks on UI elements
- Client-side cooldown mirror for responsive feel (prevents server spam)
