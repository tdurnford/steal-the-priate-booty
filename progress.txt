## 2026-03-02 — CONFIG-001: Game Configuration Module

### What was done
- Created `src/shared/GameConfig.lua` — the central shared configuration module
  containing ALL tunable game constants from the design spec.

### What's in GameConfig.lua
The module is organized into these sections:
- **Combat**: swing cooldowns, charge times, block speed, dash distance/cooldown/i-frames, per-target hit cooldown, recovery window
- **Ragdoll**: durations for light hit (1.5s), heavy hit (2.5s), blocked hit (0.5s)
- **Loot Spill**: PvP spill percentages (10%/25%/5%), min spill, bounty multiplier
- **Containers**: 6 container types (Crate, Barrel, Treasure Chest, Reinforced Trunk, Captain's Vault, Cursed Chest) with HP, yield ranges, scatter radii, spawn frequencies
- **Pickups**: despawn time (15s), max loose pickups (200), pickup radius
- **Loot Visibility**: 4 tiers based on held doubloons (none/small/medium/large)
- **Gear**: 6 cutlass tiers (Driftwood through Legendary Blade) with costs and damage values
- **Ship Tiers**: 7 tiers (Rowboat through Ghost Ship) with treasury thresholds
- **Ship System**: dock slots (24), despawn delay, raid duration/steal/cooldown
- **NPCs**: Cursed Skeleton, Ghost Pirate, Phantom Captain — full stats (HP, speed, attacks, loot, spawn budgets)
- **NPC Behavior**: SimplePath params, chase/leash/stuck detection, dormant mode, night modifiers
- **Threat**: accumulation rates, tier thresholds (Calm/Uneasy/Hunted/Cursed/Doomed), tier effects
- **Day/Night**: phase durations (Dawn 30s, Day 360s, Dusk 30s, Night 240s = 11min cycle)
- **Hazards**: Volcanic Vent, Tidal Surge, Quicksand, Rogue Wave — all timings and effects
- **Bounty**: check interval, activation threshold, clear conditions
- **World Events**: Shipwreck and Loot Surge intervals and durations
- **Notoriety/Rank**: 11 XP reward actions, 6 rank tiers with thresholds and unlocks
- **Tutorial**: distances and HP for tutorial sequence
- **Auto-Save**: 60-second interval

### Accessor functions included
- `getContainerById()`, `getGearById()`, `getShipTier()`, `getShipTierForTreasury()`
- `getRankForXP()`, `getThreatTier()`, `getLootVisibilityTier()`
- `calculateSpill()` — full PvP spill calculation with bounty support
- `canPurchaseGear()` — validation with (boolean, string?) pattern
- `isNight()`, `getNotorietyXP()`

### Lookup tables
Built at require-time for O(1) access: `ContainerById`, `GearById`, `ShipTierByNumber`, `RankByNumber`, `NotorietyXPById`

### Notes for next developer
- All values match the game design spec. If the spec changes, update this file.
- The module exports Luau types: CombatConfig, RagdollConfig, LootSpillConfig, ContainerDef, GearDef, ShipTierDef, ThreatTierDef, DayPhase, LootVisibilityTier, NotorietyXPReward, RankDef
- Follows the project's config module pattern (see .claude/skills/game-config/)
- Local toolchain (selene/rojo via aftman/rokit shims) has pre-existing issues — StyLua works via ~/.rokit/bin/stylua but selene and rojo fail with "No such file or directory" errors
- Next highest-priority features to consider: TYPES-001 (expand shared types), DATA-001 (player data schema), KNIT-001 (service scaffolding)

## 2026-03-02 — DATA-001: Player Data Schema & ProfileService Setup

### What was done
- Replaced the generic template `PlayerData` schema in `src/shared/Types.lua` with the
  full pirate game persistent data model.
- Expanded `src/server/Services/DataService.lua` with a complete getter/setter API for
  all new data fields.

### Types.lua changes
- **New export types**: `EquippedCosmetics`, `PlayerStats`, updated `PlayerData`
- **PlayerData fields**: `treasury`, `equippedGear`, `ownedGear`, `notorietyXP`,
  `tutorialCompleted`, `ownedCosmetics`, `equippedCosmetics`, `stats`, `settings`,
  `joinedAt`, `lastPlayedAt`
- **DEFAULT_PLAYER_DATA**: New players start with `rusty_cutlass` equipped and owned,
  `treasury = 0`, `notorietyXP = 0`, `tutorialCompleted = false`, empty cosmetics/stats
- **deepCopyPlayerData migration**: Handles old `money` → `treasury` migration, all
  fields use `or` fallbacks for existing saves, arrays are copied element-by-element,
  booleans use `~= nil` checks to preserve `false` values

### DataService.lua changes
- **Removed**: `UpdateMoney()`, `Client:GetMoney()` (replaced by treasury API)
- **Added treasury API**: `GetTreasury()`, `UpdateTreasury()`
- **Added gear API**: `GetEquippedGear()`, `OwnsGear()`, `PurchaseGear()`, `EquipGear()`
  — PurchaseGear validates GameConfig, ownership, and treasury balance
- **Added notoriety API**: `GetNotorietyXP()`, `AddNotorietyXP()` — fires rank change
  event when rank tier changes using `GameConfig.getRankForXP()`
- **Added tutorial API**: `IsTutorialCompleted()`, `CompleteTutorial()`
- **Added cosmetics API**: `OwnsCosmetic()`, `AddOwnedCosmetic()`, `EquipCosmetic()`
  — EquipCosmetic validates slot names and ownership, supports unequip (nil)
- **Added stats API**: `IncrementStat()`, `UpdateBiggestHaul()` — biggestHaul only
  updates if new haul exceeds previous record
- **Profile loading**: Now sets `joinedAt` on first join, updates `lastPlayedAt` on
  both load and release, sends deep copy to client
- **Client methods**: `GetData()` now returns deep copy, added `GetTreasury()`,
  `GetEquippedGear()`
- **New dependency**: Requires `GameConfig` for gear validation and rank calculation

### Backward compatibility
- Old profiles with `money` field are migrated to `treasury` via deepCopyPlayerData
- ProfileService's `Reconcile()` fills in all new fields from DEFAULT_PLAYER_DATA
- Existing client code (SettingsPanel, SoundController) is unchanged — they use
  `GetData()`, `UpdateSetting()`, and `DataChanged` which all still work

### Notes for next developer
- All persistent data fields from the spec are now defined. Session-only state
  (held_doubloons, ship_hold, threat_level, etc.) is NOT in PlayerData — that
  belongs in a future SessionState module (DATA-003).
- To add a new persistent field: update the type, add to DEFAULT_PLAYER_DATA,
  add migration in deepCopyPlayerData, add DataService method if needed.
- `PurchaseGear` uses GameConfig.GearById — if gear config changes, purchases
  still validate correctly.
- Toolchain: luau-lsp not available (not in rokit.toml), StyLua formatting passes.
- Next highest-priority features to consider: DATA-003 (session state), KNIT-001
  (service scaffolding), DATA-002 (auto-save), TYPES-001 (expand shared types)

## 2026-03-02 — DATA-003: Session-Only Player State Manager

### What was done
- Added `SessionState` type and `createSessionState()` factory to `src/shared/Types.lua`.
- Added `DEFAULT_SESSION_STATE` template alongside the existing `DEFAULT_PLAYER_DATA`.
- Created `src/server/Services/SessionStateService.lua` — a full Knit service managing
  per-player transient session state (server-authoritative, never saved to DataStore).

### Types.lua changes
- **New export type**: `SessionState` with all 15 spec fields
- **New constant**: `DEFAULT_SESSION_STATE` — default template for reference
- **New function**: `createSessionState(tutorialCompleted)` — factory that creates a
  fresh session state; sets `tutorialActive = not tutorialCompleted` and
  `tutorialStep = 1` for new players who haven't completed the tutorial,
  initializes `lastLockTime = os.clock()`

### SessionStateService.lua — complete API
- **Session lifecycle**: `_initSession(player)` on DataService.PlayerDataLoaded,
  `_cleanupSession(player)` on Players.PlayerRemoving
- **Held doubloons**: `GetHeldDoubloons()`, `AddHeldDoubloons()`, `SetHeldDoubloons()`
- **Ship hold**: `GetShipHold()`, `AddShipHold()`, `SetShipHold()`
- **Ship lock**: `IsShipLocked()`, `SetShipLocked()` — updates `lastLockTime` on lock
- **Ragdoll/combat**: `IsRagdolling()` (auto-transitions to recovery when time expires),
  `IsInRecovery()`, `StartRagdoll(duration)`, `EndRagdoll()`,
  `IsDashOnCooldown()`, `StartDashCooldown(cooldown)`
- **Per-target hit tracking**: `CanHitTarget(attacker, targetUserId)` — 2s cooldown,
  `RecordHitTarget()`, `CleanupHitTargets()` — garbage collects expired entries
- **Ship raid tracking**: `CanRaidShip(raider, ownerUserId)` — 30s cooldown per ship,
  `RecordShipRaid()`
- **Bounty**: `HasBounty()`, `SetBounty(active)`
- **Tutorial**: `IsTutorialActive()`, `GetTutorialStep()`, `SetTutorialStep()`,
  `CompleteTutorialSession()`
- **Threat level**: `GetThreatLevel()`, `AddThreat(amount)` (clamped 0-100),
  `SetThreatLevel()`, `ResetThreat()`, `GetLastLockTime()`
- **Phantom Captain**: `IsPhantomCaptainActive()`, `SetPhantomCaptainActive()`
- **Generic accessors**: `GetState()`, `GetField()`, `IsInitialized()`

### Client-visible fields (read-only via signals)
- `SessionStateChanged` signal fires to client for: `heldDoubloons`, `shipHold`,
  `shipLocked`, `threatLevel`, `hasBounty`, `tutorialActive`, `tutorialStep`
- Client-callable methods: `GetHeldDoubloons()`, `GetShipHold()`, `IsShipLocked()`,
  `GetThreatLevel()`, `HasBounty()`, `IsTutorialActive()`, `GetTutorialStep()`,
  `GetSessionSnapshot()` (returns all client-visible fields in one call)

### Server-side signal
- `StateChanged` (GoodSignal): fires `(player, fieldName, newValue)` for every field
  change — allows other services to react to state changes without polling

### Design decisions
- Session initializes after DataService.PlayerDataLoaded (not PlayerAdded) so tutorial
  completion state is available from persistent data
- `IsRagdolling()` auto-transitions to recovery state when ragdoll time expires,
  avoiding the need for a heartbeat loop
- Hit target and raid tracking use `os.clock()` timestamps with lazy cleanup,
  keeping memory bounded
- All mutation goes through typed setter methods — no direct field access from
  external services

### Notes for next developer
- To add a new session field: add it to the `SessionState` type, add default in
  `DEFAULT_SESSION_STATE` and `createSessionState()`, add getter/setter methods
  in SessionStateService, optionally add to `CLIENT_VISIBLE_FIELDS`
- The service handles studio edge case (players already in game when Knit starts)
- Roblox Studio MCP server was not available for in-game testing; tested via StyLua
  formatting check across all src files
- Selene linting skipped due to pre-existing rokit PATH issue
- Next highest-priority features: DAYNIGHT-001 (day/night cycle clock — many systems
  depend on it), DATA-002 (auto-save), KNIT-001 (service scaffolding)
