## 2026-03-02 — RANK-002: Notoriety Rank Thresholds & Unlocks

### What was done
- Created `src/server/Services/RankEffectsService.lua` — server-authoritative rank effects
  manager that applies gameplay effects based on notoriety rank.
- Added lunge and spin attack config to `GameConfig.lua` (Combat section).
- Added lunge and spin attack handlers to `CombatService.lua` (server-side).
- Added lunge and spin input handling to `CombatController.lua` (client-side).
- Modified `DoubloonService.lua` for per-player pickup radius based on rank.
- Added `DataService.NotorietyRankChanged` server-side signal for rank change events.

### All 6 rank unlocks implemented

| Rank | Name | XP | Unlock | Implementation |
|------|------|----|--------|----------------|
| 1 | Deckhand | 0 | None | Base gameplay |
| 2 | Buccaneer | 500 | Cutlass Lunge | X key: 0.3s windup → 6-stud dash + 8-stud hit, 2.0s ragdoll, 15% spill, 4s cooldown |
| 3 | Raider | 2000 | +5% Speed | WalkSpeed 16 → 16.8, applied on character spawn/rank change |
| 4 | Captain | 8000 | Cutlass Spin | C key: 0.5s windup → 360° area attack, 6-stud range, 1.5s ragdoll, 10% spill, 5s cooldown |
| 5 | Pirate Lord | 25000 | +10% Pickup Radius | Base 4 studs → 4.4 studs, per-player radius in DoubloonService |
| 6 | Dread Pirate | 75000 | Title Display | "DREAD PIRATE" BillboardGui above head, purple glow, visible to all |

### RankEffectsService features
1. **Speed bonus (Rank 3)**: Modifies Humanoid.WalkSpeed on character spawn and rank change.
   Coordinates with CombatService block speed reduction via `GetWalkSpeed()` API.
2. **Pickup radius (Rank 5)**: `GetPickupRadius(player)` returns per-player radius.
   DoubloonService queries this per collection check cycle.
3. **Title display (Rank 6)**: Server-side BillboardGui creation (replicated to all clients).
   Re-attached on character respawn. Purple text + purple PointLight glow.
4. **Unlock tracking**: Fires `UnlocksChanged` signal to client with `{lunge, spin}` booleans.
   Client RPC `GetUnlocks()` for initial sync on join.
5. **Rank change listener**: Uses `DataService.NotorietyRankChanged` server-side signal.

### CombatService changes (lunge + spin)
1. **Lunge (Rank 2)**: `LungeRequest` client signal → server validates rank, cooldown, state.
   0.3s windup delay → hit detection (8 stud range, 70° arc). Single-target like light/heavy.
   `LungeConfirm` fires to client for movement + VFX.
2. **Spin (Rank 4)**: `SpinRequest` client signal → server validates rank, cooldown, state.
   0.5s windup delay → multi-hit detection (6 stud range, 360° arc). Hits ALL targets in range.
   `SpinConfirm` fires to client for VFX.
3. **Both use `handlePlayerHitCustom()`** — shared hit logic with custom ragdoll/spill/knockback.
4. **Block speed fix**: CombatService now queries `RankEffectsService:GetWalkSpeed()` for
   the correct default speed when entering/exiting block (accounts for rank speed bonus).

### CombatController changes (client input)
1. **X key**: Lunge attack (gated by `HasLungeUnlock` from server)
   - Forward dash trail VFX (gold/orange), thrust animation, movement via LinearVelocity
2. **C key**: Spin attack (gated by `HasSpinUnlock` from server)
   - Circular particle burst VFX (gold/orange), fast spin animation
3. **State checks**: Both lunge and spin prevent other actions during execution
   (IsLunging/IsSpinning flags checked by canSwing, block, dash, primary attack)
4. **Ragdoll cleanup**: Lunge/spin state cancelled on ragdoll, character respawn

### Key bindings chosen
- **X**: Lunge — avoids conflict with F (ship lock ProximityPrompt)
- **C**: Spin — avoids conflict with E (ship deposit ProximityPrompt)
- No conflict with: WASD (move), Q (dash), G (shop), Mouse1 (swing), Mouse2 (block),
  E/F/R (ship ProximityPrompts)

### Design decisions
- **Per-player pickup radius**: DoubloonService now builds a `radiusSq` per player each
  collection check, rather than using a single global constant. Minimal perf impact since
  the check already iterates all players.
- **Multi-hit for spin**: Spin uses `performMultiHitDetection()` which returns arrays of
  all hit players/NPCs/containers. Each target gets independent ragdoll/spill/knockback.
  This makes spin powerful but fair — shorter range (6 studs) and long cooldown (5s).
- **Lunge dash is server-confirmed**: Server validates then fires `LungeConfirm` to client.
  Client applies LinearVelocity for the 6-stud forward dash. Hit detection happens
  server-side after the windup delay.
- **Speed bonus coordination**: RankEffectsService stores BASE_WALK_SPEED (16) and applies
  the bonus. CombatService queries `GetWalkSpeed()` when applying block speed reduction
  to ensure the 50% block speed is applied to the rank-boosted speed, not the base.
- **Title re-attach**: BillboardGui is re-created on CharacterAdded (not persisted on the
  old character). PointLight glow is parented to Head for proper position.

### Notes for next developer
- All lunge/spin values are in `GameConfig.Combat` — change amounts there, not in services.
- Lunge animations and spin animations use placeholder `rbxassetid://522635514` — replace
  with proper thrust/spin animations when animation assets are created.
- SoundController reuses existing sounds: lunge uses `PlayDashSound()` + `PlaySwingSound()`,
  spin uses `PlayHeavySwingSound()`. Add dedicated lunge/spin sounds when audio assets
  are available (AUDIO-001).
- The spin VFX uses a ParticleEmitter on HumanoidRootPart — replace with proper circular
  slash VFX when visual effects are polished.
- `DataService.NotorietyRankChanged` is a new server-side signal. Other services can
  listen to it for rank-based behavior (e.g., THREAT-003 rank-gated effects).
- Next priorities: NPC-003 (SimplePath patrol), NPC-006 (spawn manager), THREAT-003
  (threat effects), NPC-007 (Ghost Pirate)

## 2026-03-02 — EVENT-001: Bounty System

### What was done
- Created `src/server/Services/BountyService.lua` — server-authoritative bounty system
  that checks every 60 seconds for the player with the most held (unbanked) doubloons.
- Created `src/client/Controllers/BountyController.lua` — client-side controller that
  syncs bounty state from server, shows notifications, and exposes signals for UI.
- Integrated bounty state changes into `HudController.lua` (pulse on bounty start).

### BountyService (server) features
1. **60s check loop**: Heartbeat accumulator checks `GameConfig.Bounty.checkInterval` (60s).
   Finds the richest player with 200+ held doubloons (skips harbor players).
2. **Single bounty**: Only one active bounty at a time. Skips check if bounty is active.
3. **Skull BillboardGui**: Server-created (replicated to all clients) skull icon above
   the target's head. AlwaysOnTop, 200 stud MaxDistance, red pulsing glow layer.
4. **Clear conditions**:
   - Held doubloons drop below 100 (listens to `SessionStateService.StateChanged`)
   - 90s timeout (1s Heartbeat check)
   - Target disconnects (PlayerRemoving)
5. **Character respawn handling**: Re-attaches skull billboard on CharacterAdded.
6. **Client signals**: `BountyStarted(userId, displayName)` and `BountyEnded(userId, reason)`
   fired to all clients via Knit RemoteSignals.
7. **Server signals**: `BountyAssigned(player)` and `BountyCleared(player, reason)` for
   inter-service communication.
8. **Client RPCs**: `IsBountyActive()` and `GetBountyTargetUserId()` for late-join sync.

### BountyController (client) features
- Listens for `BountyStarted`/`BountyEnded` from BountyService
- Shows "BOUNTY ON YOU!" red warning (6s duration) to the bounty target
- Shows "Bounty placed on [name]!" orange notification (4s) to other players
- Shows bounty cleared notifications with reason-specific text
- Late-join support: queries `GetBountyTargetUserId()` on startup
- Exposes `BountyStarted`/`BountyEnded` local signals for future UI components
- Public API: `IsBountyActive()`, `GetBountyTargetUserId()`, `GetBountyTargetName()`,
  `IsLocalPlayerBounty()`

### How existing systems connect (no changes needed)
- **Double spill**: CombatService already calls `SessionStateService:HasBounty(target)`
  and passes it to `GameConfig.calculateSpill()` which applies 2x multiplier
- **Bounty XP**: NotorietyService already has `hit_bounty_target` (15 XP) and
  `survive_bounty` (25 XP) hooks that check `SessionStateService:HasBounty()`
- **Session state**: `SessionStateService:SetBounty()` already fires `hasBounty` change
  to client via `SessionStateChanged` signal

### Design decisions
- **Harbor exclusion**: Players in the harbor are skipped during the 60s check,
  preventing safe-zone abuse (hoard loot in harbor)
- **Clear on loot drop**: Listens to `SessionStateService.StateChanged` for
  `heldDoubloons` changes, immediately clears if below `clearThreshold` (100).
  This catches spills from PvP hits, not just deposits.
- **Skull as BillboardGui**: Server-side creation ensures all clients see it without
  needing per-client management. Re-attached on character respawn.
- **Single bounty constraint**: Only one bounty active at a time. The richest player
  above threshold gets it. If no one qualifies, no bounty is assigned.

### Notes for next developer
- The skull BillboardGui uses emoji text (\u{1F480}) — if Roblox doesn't render emoji
  in BillboardGui TextLabels, replace with an ImageLabel using a skull decal asset.
- SoundController:PlayUISound("bounty_warning") is called but the sound may not
  exist yet — add a bounty warning sound to SoundController when audio assets
  are available (AUDIO-001).
- Minimap bounty tracking (acceptance criteria #5) deferred to UI-005 (minimap).
  BountyController exposes `GetBountyTargetUserId()` ready for minimap integration.
- Next priorities: RANK-002 (rank unlocks), NPC-003 (SimplePath patrol),
  NPC-006 (spawn manager), THREAT-003 (threat effects)

   players at Night start, awards at Dawn to those still connected
9. **raid_ship** (10 XP): `ShipService.RaidCompleted` → awarded to raider
10. **hit_bounty_target** (15 XP): `CombatService.PlayerHitPlayer` → checks target has bounty
    (stub until EVENT-001 bounty system is implemented)
11. **survive_bounty** (25 XP): `ShipService.LockCompleted` → checks player has bounty
    (stub until EVENT-001 bounty system is implemented)

- NotorietyController tracks current XP and rank, exposes `GetXP()`, `GetRank()`,
  `GetProgressToNextRank()` — used by HudController for the indicator

- `XPAwarded` signal available for future XP toast/popup notifications
- NotorietyIndicator HUD element with rank-specific colors and icons per tier:
  Deckhand (gray ⚓), Buccaneer (green ⚔), Raider (blue 🗡), Captain (gold 👑),
  Pirate Lord (pink 🔱), Dread Pirate (purple 💀)
- XP progress bar fills toward next rank, formatted as "XP: 1.2k / 2k" in tooltip

### Design decisions
- **Pickup XP cooldown**: 5-second per-player cooldown to prevent XP spam from bulk

- **Bounty hooks**: hit_bounty_target and survive_bounty are connected but won't fire
  until EVENT-001 (bounty system) is implemented. The hooks check

- Rank calculation uses `GameConfig.getRankForXP()` which returns the RankDef

  that mirrors the server phase and exposes it to other controllers.

### DayNightService.lua — server
- **Clock**: Runs on `RunService.Heartbeat`, advancing `CycleElapsed` which wraps at
  `totalCycleDuration` (660s = 11 minutes). Uses modular arithmetic for seamless looping.
- **Phase derivation**: `getPhaseFromElapsed()` walks the ordered phase list
  (Dawn/Day/Dusk/Night) with precomputed durations from `GameConfig.DayNight`.
- **Phase transitions**: Detects when `CurrentPhase` changes and fires:
  - Server-side `PhaseChanged` signal (GoodSignal): `(newPhase, previousPhase)` — for
    other services (NPC spawning, threat, container rates, etc.)
  - Client `PhaseChanged` signal (Knit RemoteSignal): `FireAll(phase, 0, phaseDuration)`
    — broadcasts to all connected players simultaneously
- **Start phase**: Initializes at the beginning of the Day phase so players joining
  a fresh server start in daytime.
- **Server API**: `GetCurrentPhase()`, `IsNight()`, `IsDay()`, `IsDawn()`, `IsDusk()`,
  `GetPhaseProgress()`, `GetPhaseDuration()`, `GetPhaseElapsed()`, `GetCycleElapsed()`,
  `GetDurationForPhase(phase)`
- **Client-callable**: `Client:GetCurrentPhase()`, `Client:GetTimeInfo()` — GetTimeInfo
  returns `{ phase, phaseElapsed, phaseDuration, cycleElapsed, totalCycle }` for full
  initial sync

### DayNightController.lua — client
- **Initial sync**: On KnitStart, calls `DayNightService:GetTimeInfo()` to get the
  server's current phase and elapsed time (handles mid-cycle joins).
- **Phase tracking**: Listens to `DayNightService.PhaseChanged` signal and updates
  local `CurrentPhase`, `PhaseElapsed`, `PhaseDuration`.
- **Local signal**: `DayNightController.PhaseChanged` (GoodSignal) fires
  `(newPhase, previousPhase)` so other controllers (future HUD, lighting, banners)
  can react without needing a server reference.
- **Public API**: `GetCurrentPhase()`, `IsNight()`, `IsDay()`, `IsDawn()`, `IsDusk()`,
  `GetPhaseElapsed()`, `GetPhaseDuration()`, `GetPhaseProgress()`

### Design decisions
- Uses `Heartbeat` instead of `task.wait()` loop for frame-accurate timing and to
  avoid blocking the service thread
- Phase is derived from a single `CycleElapsed` counter rather than separate timers,
  making the system drift-free and easy to query
- Precomputes `PHASE_START` lookup and `PHASE_DURATIONS` from GameConfig at require-time
- Client receives elapsed-in-phase on sync so it can show accurate progress bars
- Starts at Day (not Dawn) because joining during dawn would be confusing for 30s

## 2026-03-02 — LOOT-005: Doubloon Scatter & Pickup System

### What was done
- Created `src/server/Services/DoubloonService.lua` — a server-authoritative Knit
  service that manages all doubloon pickup entities in the game world.

### DoubloonService.lua — complete implementation

#### Pickup entity creation
- Doubloon pickups are small gold cylinder Parts (coin shape) in a `workspace.Pickups`
  folder, with a PointLight for gold glow
- Each pickup stores its doubloon value as an attribute (`DoubloonValue`)
- Parts are Anchored, CanCollide=false, CanQuery=false, CanTouch=false, CastShadow=false
  for minimal physics overhead

#### Scatter system — `ScatterDoubloons(position, amount, scatterRadius)`
- Primary API entry point for other services (container breaks, PvP spills, NPC drops)
- Splits total doubloon amount into 3–15 individual pickup entities using intelligent
  splitting: small amounts get 1 pickup per doubloon, larger amounts split into
  baseValue + remainder distribution
- Each pickup lands at a random angle and distance within the scatter radius
- Pickups placed 0.5 studs above center Y for ground visibility

#### Proximity collection (server-authoritative)
- Heartbeat loop checks every 0.1 seconds (not every frame) for performance
- Builds player position list from HumanoidRootPart, then checks each pickup against
  each player using squared XZ distance (avoids sqrt)
- When player is within `GameConfig.Pickups.pickupRadius` (4 studs), the pickup is
  collected: value added to `heldDoubloons` via SessionStateService, part destroyed,
  client signal fired for VFX/SFX
- First-come-first-served: closest check wins when multiple players are near

#### Despawn system
- Every check cycle, expired pickups (>15s since creation) are removed
- Oldest-first removal ensures bounded memory

#### Global pickup cap
- Before spawning new pickups, `enforcePickupCap()` removes oldest pickups if the
  count would exceed `GameConfig.Pickups.maxLoosePickups` (200)
- Makes headroom for the incoming batch before creating new entities

#### Disconnect spill integration
- Connects to `DataService.DisconnectDoubloonSpill` signal in KnitStart
- When a player disconnects with held doubloons, scatters them at their last position
  with a 4-stud radius

#### Signals
- **Server-side**: `DoubloonService.DoubloonCollected` (GoodSignal) fires
  `(player, amount, position)` on each pickup collection — for future stat tracking,
  notoriety XP, etc.
- **Client-side**: `DoubloonService.Client.DoubloonCollected` (Knit RemoteSignal) fires
  `(pickupPosition, amount)` to the collecting player — for VFX/SFX feedback

### Public API
- `ScatterDoubloons(position, amount, scatterRadius?)` — scatter pickups at a position
- `GetActivePickupCount()` — returns current number of active pickups

## 2026-03-02 — COMBAT-005: Ragdoll Physics System

### What was done
- Created `src/shared/RagdollModule.lua` — a shared physics-based ragdoll module
  that replaces the old placeholder PlatformStanding ragdoll.
- Updated `src/client/Controllers/CombatController.lua` to use RagdollModule for
  all ragdoll visuals with knockback support.
- Updated `src/server/Services/CombatService.lua` to calculate and send knockback

- Extended `src/shared/GameConfig.lua` with knockback force values per hit type.

### RagdollModule.lua — complete implementation

#### Joint disabling system
- Iterates all Motor6D joints in the character and disables them (Enabled = false)
- Creates BallSocketConstraint replacements at each joint position using the
  motor's C0/C1 CFrame transforms
- Creates Attachment pairs on Part0/Part1 for constraint anchoring
- All joints, constraints, and attachments are tracked for clean restoration

#### Per-joint limits (exaggerated for comedy)
- 14 named joints with individual angle limits:
  - Neck: 50° upper, ±35° twist (head bobble)
  - Waist: 35° upper, ±25° twist (torso bending)
  - Shoulders: 130° upper, ±90° twist (floppy arm comedy)
  - Elbows: 130° upper, ±10° twist (wide bend)
  - Hips: 100° upper, ±35° twist (exaggerated leg flailing)
  - Knees: 100° upper, ±5° twist (dramatic leg folding)
  - Wrists/Ankles: 50° upper, moderate twist
- Slight restitution (0.3) on all joints for bouncy tumble feel
- Joints not in the named table get default limits (90°, ±45° twist)

#### Limb collision system
- Enables CanCollide on 13 body parts (Head, UpperTorso, LowerTorso, all arm
  and leg segments) so ragdolled characters interact with terrain/walls
- Original CanCollide values stored and restored on ragdoll end
- HumanoidRootPart stays non-collidable to avoid physics interference

#### Knockback impulse
- Sets AssemblyLinearVelocity on HumanoidRootPart for immediate impulse
- Adds upward kick (35% of horizontal magnitude) for exaggerated launch
- Adds angular velocity perpendicular to knockback direction for comedic spin
  (capped at 15 rad/s to prevent excessive rotation)

#### Humanoid state management
- PlatformStanding = true to prevent auto-stand during ragdoll
- Disables GettingUp and Running states to prevent Humanoid interference
- On restore: re-enables states, sets PlatformStanding = false, triggers
  GettingUp state for smooth stand-up animation

### CombatController.lua changes
- **Removed**: Old `applyRagdollVisual()` function (PlatformStanding placeholder)
- **Added**: RagdollModule import and usage
- **Updated onRagdollTrigger**: Now accepts `knockbackVelocity: Vector3?` parameter,
  calls `RagdollModule.enable(character, knockbackVelocity)` instead of
  PlatformStanding, schedules `RagdollModule.disable()` after duration
- **Added RagdollCleanupThread**: Tracks the pending disable task so overlapping
  ragdolls cancel the old cleanup before starting a new one
- **Added CharacterRemoving handler**: Calls `RagdollModule.cleanup(character)` and
  resets all ragdoll state (IsRagdolled, RagdollCleanupThread) on respawn/death

### Design decisions
- Module is in src/shared/ for potential future server-side NPC ragdoll use
- BallSocketConstraint chosen over HingeConstraint for more chaotic tumbling

- Knockback is computed server-side and sent to client so it can't be exploited
- Upward kick and angular velocity are computed from knockback magnitude, meaning
  stronger hits automatically produce more dramatic tumbles
- Joint limits are intentionally loose (especially shoulders/elbows) so arms flail
  dramatically during ragdoll — this is the "core humor element" from the spec

### ShipService.lua — complete system

#### Dock slot management
- Dock slots defined as Parts in `workspace.ShipDockPoints` with `SlotIndex`
  attribute (1-24).
- `getDockPoints()` fetches all dock point Parts, sorted by SlotIndex.
- `getAvailableDockSlot()` returns first unoccupied slot.

- `DockedShips: { [Player]: ShipEntry }` tracks player → ship mapping.

#### Ship spawning
- Triggers on `DataService.PlayerDataLoaded` signal (not PlayerAdded), because
  ship tier requires treasury balance from persistent data.
- `spawnShip(player)` calculates tier via `GameConfig.getShipTierForTreasury()`,
  finds available dock slot, creates placeholder model, registers in both
  DockedShips and DockSlotOccupancy registries.
- Handles studio edge case (players already in game when Knit starts).

#### Ship model creation (placeholder)
- 7 ship tier appearances defined in `SHIP_APPEARANCE` table (rowboat through
  ghost_ship) with size, color, and material per tier.

  OwnerName.
- Mast part for tiers >= 2 (Sloop and above).
- Ghost Ship (tier 7) gets spectral PointLight and ParticleEmitter glow.
- BillboardGui with owner name and ship tier name label (visible from 60 studs).
- Ships parent to `workspace.DockedShips` folder.

#### Ship despawning
- On PlayerRemoving: removes from DockedShips but keeps in DockSlotOccupancy,
  starts 30-second delayed despawn via `task.delay`.
- `PendingDespawns: { [number]: thread }` tracks userId → despawn thread.
- If player rejoins before timer expires, pending despawn is cancelled and old
  ship is cleaned up so a fresh one spawns.
- `BindToClose` immediately destroys all ship models and cancels pending
  despawns on server shutdown.

#### Ship tier recalculation
- `RecalculateShipTier(player)` checks treasury against
  `GameConfig.getShipTierForTreasury()`, and if tier changed: destroys old model,
  creates new model at same dock position, fires tier change signals.
- Called by future SHIP-002/003 when treasury changes (deposit+lock, gear
  purchase, cosmetic purchase).

## 2026-03-02 — THREAT-001: Threat Level Tracking & Accumulation

### What was done
- Created `src/server/Services/ThreatService.lua` — server-authoritative threat
  accumulation service that drives the PvPvE tension mechanic.

### ThreatService.lua — complete implementation
- **Time-based accumulation**: +5 per minute since last lock, computed per-second
  from `GameConfig.Threat.timeRate / 60`. Uses per-player `TimeAccumulators` that
  accumulate delta-time and fire every 1 second.
- **Held doubloons check**: Every 30 seconds (configurable via
  `GameConfig.Threat.heldDoubloonsInterval`), calculates
  `floor(heldDoubloons / 50) * 1` threat. Per-player `HeldDoubloonTimers` track
  the interval.
- **Container break hook**: Subscribes to `ContainerService.ContainerBroken` signal.
  Adds +2 threat (from `GameConfig.Threat.containerBreakGain`) to the breaking
  player.
- **NPC kill hook**: `OnNPCKilled(player)` method exposed for future NPC services.
  Adds +3 threat (from `GameConfig.Threat.npcKillGain`).
- **Danger zone check**: `IsInDangerZone(player)` returns false (placeholder until
  ZONE-001). When implemented, +3/min in danger zones via `DangerZoneAccumulators`.
- **Night multiplier**: All threat gains multiplied by 1.5x during Night phase
  via `getNightMultiplier()` using `DayNightService:IsNight()`.
- **Harbor pause**: `IsInHarbor(player)` checks for a workspace `HarborZone` Part
  using AABB bounding box detection. All periodic and event-driven threat gains
  skip players inside the Harbor. When HARBOR-001 places a `HarborZone` part in
  workspace, the pause activates automatically.
- **Central `AddThreatWithMultiplier` method**: All threat sources route through
  this, which applies night multiplier and harbor check before delegating to
  `SessionStateService:AddThreat()`.

### Integration with existing services
- **ContainerService**: Already fires `ContainerBroken(containerEntry, attackingPlayer)`
  signal — ThreatService subscribes to it. No changes to ContainerService needed.
- **ShipService**: Already handles threat reduction:
  - `DepositAll`: calls `SessionStateService:AddThreat(player, -25)`
  - `LockShip`: calls `SessionStateService:SetThreatLevel(player, 0)`
  No changes to ShipService needed.
- **SessionStateService**: Already has full threat API (`AddThreat`, `SetThreatLevel`,
  `GetThreatLevel`, `ResetThreat`, `GetLastLockTime`) and replicates `threatLevel`
  to the client via `SessionStateChanged` signal. No changes needed.
- **DayNightService**: Used for night multiplier check. No changes needed.
- **Auto-discovery**: Server `Main.server.lua` auto-loads all services in the
  Services folder. ThreatService is picked up automatically.

### Cleanup
- Per-player timer tables (`HeldDoubloonTimers`, `TimeAccumulators`,
  `DangerZoneAccumulators`) are cleaned up on `Players.PlayerRemoving`.
- Heartbeat tick runs every 1 second to keep CPU overhead minimal while providing
  responsive threat updates.

## 2026-03-02 — SHIP-004: Ship Raiding System

### What was done
- Implemented server-authoritative ship raiding in `src/server/Services/ShipService.lua`
- Added client-side raid interaction with progress bar in `src/client/Controllers/ShipInteractionController.lua`
- Added raid sound effects in `src/client/Controllers/SoundController.lua`

### ShipService.lua changes (server)
- **New client signals**: `RaidCompleted` (raider gets stolen amount), `RaidAlert` (owner gets warning), `RaidInterrupted` (raider gets cancellation reason)
- **New server signal**: `RaidCompleted` (for inter-service use)
- **New RPCs**: `Client:StartRaid(targetSlotIndex)` — validates everything, starts 3s timer; `Client:CancelRaid()` — client-initiated cancel
- **New API**: `CancelRaid(player, reason)` — cancel a raid (used by ragdoll interrupt, disconnect, movement); `IsRaiding(player)` — check if player is raiding
- **Active raid tracking**: `ActiveRaids` table tracks per-raider state (target slot, owner, start position, timer thread)
- **Validation chain**: Can't raid own ship, ship must be unlocked, ship hold > 0, 30s per-ship cooldown, not ragdolled, proximity check (20 studs)
- **Completion logic**: Steals 25% of hold (math.ceil, min 1), adds to raider's heldDoubloons, records cooldown via SessionStateService
- **Interrupt on ragdoll**: Listens to `SessionStateService.StateChanged` for `isRagdolling=true` to auto-cancel raids
- **Interrupt on movement**: Heartbeat loop checks raider hasn't moved >10 studs from start position (safe table iteration with collect-then-cancel pattern)
- **Disconnect cleanup**: `onPlayerRemoving` cancels any active raid
- **BindToClose**: Cancels all active raids on server shutdown
- **Race condition fix**: Raid entry registered in `ActiveRaids` BEFORE timer thread is created; thread assigned to entry after creation; `CancelRaid` handles nil thread gracefully

### ShipInteractionController.lua changes (client)
- **Raid prompts on OTHER players' ships**: Creates "Raid Ship" ProximityPrompt (R key) on every non-owned docked ship hull
- **Raid prompt lifecycle**: Created on ShipSpawned, removed on ShipDespawned, recreated on ShipTierChanged, batch-created on late join via GetAllDockedShips
- **Progress bar**: BillboardGui above player's head with red fill bar + "RAIDING..." label, updated every Heartbeat
- **Raid flow**: Prompt triggers → server StartRaid RPC → on success, show progress bar + play start SFX → on complete, play complete SFX + show notification → on interrupt, play interrupt notification
- **Raid alert**: When RaidAlert signal fires (someone raiding YOUR ship), shows red notification + alarm SFX
- **Cleanup**: Progress bar GUI cleaned up on complete, interrupt, or cancel

### SoundController.lua changes
- **New sounds**: `raidStart` (tense interaction), `raidComplete` (loot stolen chime), `raidAlert` (urgent alarm horn)
- **New methods**: `PlayRaidStartSound()`, `PlayRaidCompleteSound()`, `PlayRaidAlertSound()`
- Sound asset IDs are placeholder (reusing existing Roblox IDs); proper raid-specific audio can be swapped in during AUDIO-001

### Key design decisions
- Server manages the 3s timer (not client) to prevent exploits
- Raid prompts are client-created (like deposit/lock prompts) but server validates everything
- Raiding IS allowed in Harbor (per spec: "No PvP in Harbor, but raiding IS allowed in Harbor")
- The server does NOT check inHarbor for raid validation — only CombatService checks harbor for PvP
- Movement interrupt threshold is 10 studs (generous — allows small adjustments but prevents walk-away exploits)
- Heartbeat movement check uses collect-then-cancel pattern to avoid table modification during iteration

## 2026-03-02 — NPC-001: Cursed Skeleton NPC Base Setup

### What was done
- Created `src/server/Services/NPCService.lua` — a complete server-authoritative NPC
  management service for Cursed Skeleton entities.
- Extended `src/server/Services/CombatService.lua` to detect and damage NPCs via the
  existing swing hit detection system.

### NPCService.lua — full feature set

**NPC Entity Creation**:
- Placeholder Cursed Skeleton model with Humanoid, body parts (torso, head, arms, legs),
  cutlass weapon, and eerie green eye glow (PointLight)
- Humanoid configured: 20 HP, 0.9x player walk speed, health bar always visible at 30 studs
- All body parts have `NPCId` and `NPCType` attributes for hit detection lookup
- Models created in `workspace.NPCs` folder

**AI State Machine**:
- States: `idle`, `patrol`, `chase`, `attack_slash`, `attack_lunge`, `flinch`, `dead`
- **Patrol**: Wanders within 15 studs of spawn position, picks random points every 5s
- **Chase**: Activates when player enters aggro range (40 studs), follows via
  `Humanoid:MoveTo()`. Re-evaluates target if a closer player appears.
  Respects leash distance (80 studs from spawn) and Harbor safe zone.
  Drops chase if target leaves 1.5x aggro range (hysteresis).
- **Attack selection**: Prefers lunge when target is 3-8 studs away and lunge is off cooldown.
  Falls back to slash attack otherwise.

**Slash Attack**:
- 0.8s windup (NPC frozen, facing target)
- Hit check at end of windup against target player
- 2s cooldown between slashes
- On hit: 2.0s ragdoll, 20% loot spill (uses `GameConfig.calculateSpill`)
- Respects block (reduced to 0.5s ragdoll, 5% spill, no knockback)
- Knockback velocity calculated from NPC→target direction

**Lunge Attack**:
- 0.5s crouch telegraph (NPC frozen)
- 0.3s dash forward toward target position
- Hit check at end of dash
- 5s cooldown between lunges
- Same hit effects as slash (ragdoll + spill)

**NPC Hit Reception**:
- `DamageNPC(npcId, damage, attackingPlayer)` — applies gear damage
- 0.2s flinch on hit (pauses AI, then resumes chase/patrol)
- Health bar updates via Humanoid.Health

**Death & Loot Drop**:
- On death: scatters `carriedDoubloons + random(30-80)` bonus doubloons
- Calls `ThreatService:OnNPCKilled(player)` for +3 threat
- Sends client signals for death VFX
- 2s delay before model removal (allows death VFX to play)

**Respawn System**:
- 90-second respawn timer per NPC
- Respawns at original spawn position in same zone
- Respawn queue processed on Heartbeat

**Night Modifiers**:
- Speed bonus: +25% walk speed during night (via `GameConfig.NPCBehavior.nightSpeedBonus`)
- Aggro range bonus: +30% during night (via `GameConfig.NPCBehavior.nightAggroRangeBonus`)
- Updated per-frame via Heartbeat

**Spawn System**:
- Reads `workspace.NPCSpawnPoints` folder for spawn point Parts
- Each Part can have `Zone` (string) and `NPCType` (string, default "skeleton") attributes
- Falls back to spawning 3 test skeletons near origin if no spawn points exist

**Client Signals**:
- `NPCSpawned(npcId, npcType, position)` — for all players
- `NPCDamaged(npcId, hpFraction)` — for VFX feedback
- `NPCDied(npcId, npcType, position)` — for death VFX
- `NPCAttack(npcId, attackType, targetPosition)` — for attack SFX/VFX
- `NPCFlinch(npcId)` — for flinch VFX

**Server Signals**:
- `NPCService.NPCDied(npcEntry, killedByPlayer)` — for inter-service hooks
- `NPCService.NPCDamaged(npcEntry, damage, attackingPlayer)` — for tracking

## 2026-03-02 — THREAT-003: Threat Level Effects — Tiers 1-3 (Calm, Uneasy, Hunted)

### What was done
- Created `src/server/Services/ThreatEffectsService.lua` — server-authoritative threat tier
  effects manager that listens to SessionStateService.StateChanged for threat level changes,
  detects tier transitions, and activates/deactivates gameplay effects per player.
- Created `src/client/Controllers/ThreatEffectsController.lua` — client-side controller for
  threat tier VFX and audio (eerie ambient sound for Uneasy, fog vignette for Hunted,
  tier transition notifications).
- Modified `src/server/Services/NPCService.lua` — added forced target support, per-player
  aggro range overrides, threat-based NPC speed bonuses, bonus NPC spawn/despawn methods.

### Threat tier effects implemented

| Tier | Range | Server Effects | Client Effects |
|------|-------|----------------|----------------|
| Calm | 0-19 | None | None |
| Uneasy | 20-39 | NPCs within 60 studs gain +10% speed | Faint eerie ambient audio loop |
| Hunted | 40-59 | Bonus skeleton spawns targeting player; NPCs use 60-stud aggro range | Fog vignette overlay + louder eerie audio + notification |

### ThreatEffectsService features
1. **Tier tracking**: Listens to `SessionStateService.StateChanged` for `threatLevel` changes.
   Computes tier from `GameConfig.getThreatTier()` and detects transitions up/down.
2. **Uneasy speed bonus**: `GetSpeedBonusNearPosition(pos)` returns +10% bonus if any
   Uneasy+ player is within 60 studs. NPCService queries this during each AI tick.
3. **Hunted aggro override**: `GetAggroRangeForPlayer(player)` returns 60 studs for
   Hunted+ players (vs default 40). NPCService checks per-player in `findClosestTarget()`.
4. **Bonus skeleton**: Spawns one Cursed Skeleton with `forcedTarget` set to the Hunted
   player. No leash distance, no aggro range limit — it always chases the target.
   Respawns 10s after death if player still Hunted+. Despawns when threat drops.
5. **Client signal**: `ThreatTierChanged(tierId, tierName, isUpward)` fires per player on
   tier transitions. Late-join sync via `GetThreatTier()` client RPC.
6. **Cleanup**: Despawns bonus NPCs and clears tracking on player disconnect.

### NPCService changes
1. **`forcedTarget` field**: NPCEntry can have a forced target player. When set, the NPC
   skips normal target selection and always chases that player. Leash distance and aggro
   range checks are bypassed for forced-target NPCs.
2. **`isBonusNPC` field**: Bonus NPCs skip the normal respawn queue on death. Their
   lifecycle is managed by ThreatEffectsService.
3. **`SpawnBonusSkeleton(pos, player)`**: Spawns a skeleton with forcedTarget and isBonusNPC.
4. **`DespawnBonusNPC(npcId)`**: Immediately despawns a bonus NPC (no respawn queue).
5. **Per-player aggro range**: `findClosestTarget()` now checks
   `ThreatEffectsService:GetAggroRangeForPlayer()` per player to allow Hunted players to
   be detected at 60 studs instead of the base 40.
6. **Threat speed bonus**: `getEffectiveSpeed(npcPosition)` now queries
   `ThreatEffectsService:GetSpeedBonusNearPosition()` for additional speed multiplier
   from nearby Uneasy+ players.

### ThreatEffectsController features
1. **Eerie ambient audio**: Looped Sound in SoundService with volume fade-in/out.
   Uneasy = 0.08 volume (barely audible), Hunted = 0.15 volume (more noticeable).
2. **Fog vignette**: ScreenGui with fullscreen Frame + UIGradient for radial vignette
   (dark edges, transparent center). Pulses between 0.55-0.70 transparency on a 3s sine
   loop for unsettling feel. Fades in on Hunted, fades out on tier decrease.
3. **Tier notifications**: Shows toast notification via NotificationController on each
   tier transition with tier-specific color and message.
4. **Late-join sync**: Queries server for current tier on start and applies effects.
5. **Respawn sync**: Re-syncs tier from server after character respawn.

### Config values used (already in GameConfig.Threat)
- `uneasySkeletonSpeedBonus` = 0.10 (+10%)
- `huntedBonusSkeletonCount` = 1
- `huntedAggroRange` = 60 studs
- `cursedNpcSpeedBonus` = 0.20 (+20%, pre-wired for THREAT-004)

### Design decisions
- **Per-NPC speed query**: ThreatEffectsService checks all Uneasy+ players each time
  NPCService queries speed. This is O(uneasy_players) per NPC per frame, but typically
  very few players are Uneasy at once, so performance impact is minimal.
- **Forced target lifecycle**: Bonus skeletons respawn 10s after death (not 90s normal
  respawn) to maintain pressure. They despawn immediately when threat drops.
- **Vignette pulsing**: Sine wave oscillation gives an organic "breathing" feel rather
  than a static overlay, creating unease without being annoying.
- **Pre-wired Cursed/Doomed**: GetSpeedBonusNearPosition already returns the Cursed +20%
  bonus for Cursed+ tier players, ready for THREAT-004. Only the additional Cursed/Doomed
  effects (trap containers, Phantom Captain, ghostly footprints, dark aura) remain.

  with a proper ambient drone/whisper track when audio assets are available (AUDIO-001).
